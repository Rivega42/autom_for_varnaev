# 17 · Видеозапись (NVR Frigate) + выборочный анализ фрагментов

> Эпик. Заказчик попросил **видеозапись**. По исходному ТЗ запись — функция АУРА
> (CLAUDE.md §5), но **АУРА не пишет**. Добавляем запись в наш контур так, чтобы
> **не дублировать** уже построенную аналитику и не жевать 20 потоков 24/7.

## 1. Идея (она же — наш план v1)

Это ровно то, что закладывалось в `CLAUDE.md` §4:
> «Триггер аналитики — **по расписанию** (наш планировщик). Источник видео —
> RTSP-поток **ИЛИ видеофайл/фрагмент**.»

Делим систему на два независимых слоя:

- **Запись (NVR)** — отдельный сервис **Frigate**: пишет весь таймлайн (по
  движению / 12 ч), хранит 60–90 дней, даёт веб-плейбек (таймлайн, экспорт).
- **Аналитика (наш контур)** — НЕ работает 24/7. По **расписанию** или по
  **запросу** берёт **кусочек** записи (вырезанный за интервал), прогоняет наш
  пайплайн (халат / протирание / присутствие / …) и кладёт события + скриншоты в
  журнал, со ссылкой обратно на таймлайн Frigate.

Frigate **не заменяет** нашу аналитику: его собственная детекция нужна только
чтобы триггерить запись. Халат, протирание, покрытие зон, присутствие, пороги
датчиков, журнал, «стена» — остаются у нас.

## 2. Архитектура

```
Камеры (RTSP 720p, до 20) ──► go2rtc (один забор потока, раздаёт потребителям)
                                 ├──► наша video-analytics (по требованию)
                                 └──► Frigate ──► запись по движению + ретеншен + плейбек-UI
                                                  └── API: вырезать клип за [start..end]

Триггер (расписание / событие / оператор / АУРА-v2)
      │
      ▼
nvr-connector ──1─► Frigate API: GET /api/<камера>/start/<ts>/end/<ts>/clip.mp4
              ──2─► сохраняет клип на общий том (/clips)
              ──3─► POST /api/v1/analysis-tasks {source_type:"file", source_ref, camera_id}
                          │
                          ▼
                 наш воркер ──► только включённые проверки (тумблеры камеры:
                                pose/actions/uniform/coverage) ──► события + скриншоты
                          │
                          ▼
                 журнал событий ──► (бонус) закладка ↔ время в Frigate
```

## 3. Что переиспользуем (уже готово)

| Кусок | Где |
|---|---|
| File-анализ фрагмента | `POST /api/v1/analysis-tasks` с `source_type=file` (см. «стена роликов») |
| Планировщик «по расписанию» | сервис `scheduler` + `GET/POST /api/v1/schedules` |
| Пайплайн распознавания | `video-analytics` (позы/действия/халат/покрытие) |
| Селективность по камере | тумблеры `analytics: {pose, actions, uniform, coverage}` (PATCH /cameras) |
| Журнал событий + скриншоты | `events`, `artifacts`, `GET /events`, `GET /artifacts` |
| Разъём для внешнего триггера | `POST /integration/analysis-tasks` (заглушён, готов к включению в v2) |

## 4. Что добавляем (новое)

1. **Frigate** в стек — `docker-compose.nvr.yml` (оверлей): сервис `frigate`,
   запись по движению/детекции, ретеншен 60–90 дней, том записей, веб-UI.
2. **nvr-connector** — тонкий сервис/эндпойнт: по триггеру вырезает клип из
   Frigate за интервал, кладёт на `/clips`, создаёт file-задание воркеру с
   тумблерами камеры. Источник фрагмента — Frigate (раньше в плане это была
   «АУРА v2»). `# СТЫК-АУРА (v2)`: тот же коннектор сможет брать фрагмент от АУРА.
3. **Расписания анализа** — привязка cron к коннектору (напр. «кухня, каждый час,
   последние 10 мин → халат+протирание»).
4. **Закладки** (бонус) — связать наши события с таймлайном Frigate (UI/ссылка),
   чтобы оператор из журнала прыгал в нужную секунду видео.

## 5. Параметры объекта (от заказчика)

- До **20 камер**, поток **720p** (~1.5 Мбит/с).
- Запись **по движению** (вероятно) или 12 ч/сутки.
- Хранение **60–90 дней**.
- Плейбек-UI — **желателен** (есть у Frigate из коробки).
- АУРА **не пишет** — запись делаем мы.

## 6. Хранилище (расчёт, 720p ≈ 1.5 Мбит/с ≈ 0.66 ГБ/ч на камеру)

| Режим | В сутки (20 кам) | За 60 дней | За 90 дней |
|---|---|---|---|
| По движению (~6–8 акт. ч/сутки) | ~90–120 ГБ | ~5.4–7 ТБ | ~8–11 ТБ |
| 12 ч/сутки непрерывно | ~160 ГБ | ~9.5 ТБ | ~14 ТБ |

Закладывать **~12–16 ТБ полезного** места, лучше **RAID1/RAID5** (сырых дисков в
1.5–2× больше) ради надёжности доказательной базы.

## 7. Железо под Frigate

- **Запись по движению** (пиксельная детекция) — лёгкая, хватит CPU.
- Запись «по **человеку**» (умнее, меньше мусора) — нужна объектная детекция:
  **Coral TPU** (~$60, тянет много камер) или GPU. На 20×720p — один Coral.

## 8. Этапы

1. **PoC** (этот эпик): Frigate пишет тестовый поток → коннектор по расписанию
   вырезает кусок → наш воркер выдаёт «халат/протирание» по нему → события.
2. **MVP**: реальные камеры, ретеншен, плейбек-UI, расписания на боевых камерах,
   закладки событие↔запись.
3. **Прод**: дисковый массив/RAID, мониторинг места, (опц.) Coral TPU, бэкап.

## 9. Граница v1/v2 и АУРА

Запись (Frigate) — наш контур, АУРА не касается. Коннектор берёт фрагмент из
Frigate; в v2 он же сможет принимать фрагмент/команду от АУРА через уже
заглушённый `POST /integration/analysis-tasks` — без переписывания. Наружу контур
по-прежнему отдаёт **события**, а не сырьё (симметрия с датчиками, CLAUDE.md §4).
