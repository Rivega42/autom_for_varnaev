# 07 · Видеоаналитика (перенос рабочего PoC)

> **Важно для Кода: движок аналитики НЕ пишется с нуля.** Существует рабочий,
> проверенный прототип — браузерное приложение `motion-log.html`. Этот документ
> фиксирует его поведение как **эталон**, который переносится в Python-воркер
> `services/video-analytics`. Сам файл PoC кладётся в
> `services/video-analytics/reference/motion-log.html` и служит источником логики
> (пороги, окна, формулы). Задача E4 — **порт**, а не изобретение.

---

## 1. Что уже работает (PoC)

Браузерное приложение: камера → MediaPipe → движок распознавания → журнал с
метками времени и экспортом в `.txt`/`.json`. Проверено: на одной камере крутится
в реальном времени на CPU.

- **Модель:** MediaPipe **Tasks Vision `PoseLandmarker`**, одна модель, **33 точки
  тела** (голова, плечи, локти, запястья, бёдра, колени, лодыжки, стопы).
- **Лево/право — анатомические** (относительно человека); кадр зеркалится как
  селфи, скелет зеркалится в коде один раз (не дважды).
- **`visibility`-проверка**: ноги/колени срабатывают только когда реально в кадре.
- **Ползунок чувствительности** двигает все пороги разом; тумблеры включают/
  выключают группы детекторов.

## 2. События движка (эталон для порта)

### 2.1 Простые события (смена позы; пороги + гистерезис + антидребезг)
- Руки: «Поднята правая/левая рука — над головой / вверх / в сторону», «Опущена
  рука», «Подняты обе руки».
- Ноги: «Поднято левое/правое колено».
- Голова: «повёрнута влево/вправо», «прямо».
- Корпус: «Приседание» / «Подъём из приседа», «Наклон корпуса влево/вправо».

Реализация: пороги по высоте запястья/колена с гистерезисом и антидребезгом.

### 2.2 Составные действия (скользящее окно ~1.5 с, с длительностью)
- **Протирание стола**: «Протирание поверхности двумя руками/левой/правой, N с».
  Счётчики разворотов рук (`lRev>=3 && rRev>=3` для двух рук; одна рука при
  бездействующей второй — одноручный режим). Развороты засчитываются только пока
  кисть в ROI-зоне «стол» (если зоны заданы) — реализовано в `WipingDetector` +
  `CompositeActionAnalyzer.process(..., table_polygons)`.
- «Машет рукой» → «Помахивание завершено (N с)».
- «Хлопок в ладоши».
- «Ходьба на месте» → «Остановился (ходьба N с)».

Реализация: скользящее окно, подсчёт разворотов движения по горизонтали
(протирание/махи) и чередования колен (ходьба).

### 2.3 ROI-зоны и процент покрытия
- Зоны размечаются **вручную** (типы: стол / пол / окно). Авторазметка из PoC
  убрана — оставлена ручная.
- **% покрытия** считается раз в секунду: heat-слой кладётся на уменьшенную копию
  кадра (≈200 px), берётся доля площади полигона зоны, реально закрашенная
  движением (закрашенные пиксели / все пиксели зоны). Считается по каждому типу
  зоны отдельно.

### 2.4 Распознавание формы («белый халат»)
- **Способ из PoC — цветовая эвристика по торсу** (без отдельной модели): по
  точкам плеч (11/12) и бёдер (23/24) строится четырёхугольник торса, читаются
  пиксели кадра внутри него, считаются средняя **яркость** и **насыщенность**.
  Белый халат = высокая яркость + низкая насыщенность.
- Использование: индикатор, событие в журнал («сотрудник без формы») или
  **условие** (засчитывать уборку только если человек в форме).
- **Правило-алерт (#272):** если при видимом торсе халат не распознан дольше
  `ANALYTICS_UNIFORM_MIN_S` секунд (по умолчанию 5) — событие `uniform_violation`
  (WARNING) со стоп-кадром, раз на эпизод; при возврате халата эпизод
  сбрасывается. Тумблер — `cameras.analytics.uniform`. Это эвристика, не
  обученный детектор СИЗ (зональная привязка и реальный детектор — #105/follow-up).
- **Честные ограничения:** путается на фоне белой стены/нержавейки, при
  пересвете; не отличает халат от белого фартука/футболки. Для сигнала — годится,
  для строгого контроля — слабо.
- **Продакшн-альтернатива (v2, отдельный issue):** обученный детектор формы/СИЗ
  (YOLO-класса) или классификатор по кропу торса. Тяжелее (нужны данные и
  компьют), зато устойчив к фону и свету. В v1 **не** делаем.

### 2.5 Запретная зона (#299)
- Зона типа `forbidden` (ROI-полигон, как обычные зоны). Тумблер —
  `cameras.analytics.presence`.
- Репрезентативная точка человека — **середина бёдер** (если видны), иначе
  плеч; проверяется вхождение в полигон (point-in-polygon).
- Вход → событие `forbidden_zone_entry` (WARNING) со стоп-кадром, **раз на
  эпизод** по зоне (сброс при выходе/исчезновении человека).
- Ограничение: это присутствие по позе, без трекинга конкретных людей.
- **Рабочая зона `work` (#302):** присутствие точки человека в ней → событие
  `presence_detected` (INFO, раз на эпизод). Это сигнал присутствия в журнал;
  оконный алерт «нет присутствия в требуемом окне» (`presence_missing`) — #300.

## 3. Что переносится в Python (E4)

- Пайплайн именуется **`pose_v1`** (значение поля `analysis_tasks.pipeline`).
- Модель: тот же `PoseLandmarker`; для плотности можно `pose_landmarker_lite`.
- Источник кадров — параметр (`stream` через media-gateway | `file` с тома).
- На выходе воркер шлёт **события** (не видео) в `log-service` и сохраняет
  **артефакты**: скриншот-кадр + при необходимости keypoints/coverage в JSON
  (см. `04_DATA_MODEL.md`, виды артефактов `screenshot` / `keypoints`).
- Конфигурация камеры (ROI-зоны + ракурс/viewpoint), которую PoC экспортировал в
  JSON, становится записью в БД (таблицы `cameras` / `camera_zones`, см. `04`).

## 4. Границы и масштабирование (важно — что НЕ тащим в v1)

В прошлых проработках был намечен путь масштабирования на 50–200 камер: пул
воркеров, GPU-батчинг (16–32 кадра), очередь (Kafka / Redis Streams / NATS),
снапшоты в объектное хранилище (S3/MinIO). **Для нашего объекта (один сервер,
немного камер) это избыточно и в v1 НЕ внедряется.** Держим просто:

- кадры через media-gateway, инференс на CPU при 5–8 fps;
- события — прямым внутренним REST в `log-service` (без брокера очередей);
- артефакты — на общий том (без S3).

GPU-батчинг и очередь — задел на будущее, заводится отдельным решением, если число
камер вырастет. В v1 не усложняем.
