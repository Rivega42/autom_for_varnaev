# 11 · Руководство заказчика

Этот документ — единая точка входа для заказчика: что это за система, как её
развернуть на сервере объекта, как завести помещения/датчики/камеры, как
настроить видеоаналитику, где смотреть данные и как обращаться к API. Документы
с деталями указаны ссылками; здесь — практический путь «от коробки до работы».

> Терминология. **Наш контур** — это ПО мониторинга помещений (датчики,
> видеоаналитика, журнал событий, дашборды), которое мы поставляем. **АУРА** —
> отдельный продукт заказчика (контроль производства, видеозапись и т.п.). В
> версии 1 наш контур работает **самостоятельно** и с АУРА не обменивается
> (разъёмы интеграции есть, но выключены — см. §10).

---

## 1. Что делает система

В каждом помещении объекта стоят датчики (температура воздуха, влажность,
бесконтактная ИК-температура поверхности) и камеры. Все данные сводятся на **один
сервер на объекте**, где в контейнерах работает наш контур:

- **Сбор показаний датчиков** по MQTT, накопление в базе данных (временные ряды).
- **Видеоаналитика** по видеопотоку камер: распознавание поз и действий человека,
  расчёт % покрытия рабочих зон (ROI), эвристика «спецодежда / белый халат».
- **Единый журнал событий**: и датчики, и видеоаналитика порождают **события**
  с человекочитаемым сообщением для оператора (например «В холодильной камере
  температура выше нормы», «Покрытие зоны стола — 63%»).
- **Дашборды Grafana** для просмотра рядов и событий.
- **Веб-интерфейс настройки** видеоаналитики (камеры, функции, рабочие зоны).

**Чего система НЕ делает** (это зона АУРА, мы не реализуем): видеозапись,
технологические карты, контроль производства, программу производственного
контроля. Наружу контур отдаёт **события**, а не сырьё (не видеозапись и не сырые
ряды датчиков).

---

## 2. Архитектура в двух словах

Система — это набор контейнеров Docker на одном сервере, разделённых на две сети:

| Компонент | Назначение | Доступ снаружи |
|---|---|---|
| `api-gateway` | единственный внешний REST-вход + веб-GUI настройки | **да** (порт 8000) |
| `grafana` | дашборды | **да** (порт 3000) |
| `db` (TimescaleDB) | база: ряды датчиков, события, задания, справочники | нет |
| `mqtt-broker` (Mosquitto) | приём показаний от узлов датчиков | только в LAN объекта |
| `ingest-sensors` | MQTT → база, контроль «тишины» узлов | нет |
| `media-gateway` (go2rtc) | приём RTSP/ONVIF камер, кадр-превью | нет |
| `video-analytics` | распознавание (MediaPipe), события, скриншоты | нет |
| `scheduler` | запуск видеоанализа по расписанию | нет |
| `log-service` | единый журнал событий | нет |
| `migrate` | разовое применение миграций БД при старте | нет |

**Наружу сервера опубликованы только два порта: `api-gateway` (8000) и Grafana
(3000).** Всё остальное доступно лишь во внутренней сети контейнеров. Подробнее —
[`docs/01_ARCHITECTURE.md`](01_ARCHITECTURE.md) и [`docs/02_NETWORK.md`](02_NETWORK.md).

## 3. Развёртывание (кратко)

Полная пошаговая инструкция — [`docs/10_DEPLOYMENT.md`](10_DEPLOYMENT.md). Кратко:

```bash
# 1. Получить код
git clone <repo-url> monitoring && cd monitoring

# 2. Настроить окружение (секреты!)
cp .env.example .env
#   обязательно задать: POSTGRES_PASSWORD, POSTGRES_RO_PASSWORD, API_KEY,
#   GF_SECURITY_ADMIN_PASSWORD (не оставлять change-me)

# 3. Положить модель MediaPipe (иначе видеоаналитика не стартует)
#   models/pose_landmarker.task  (см. models/README.md)

# 4. Описать камеры объекта в media-gateway/go2rtc.yaml (см. §6)

# 5. Поднять стек
docker compose up -d
docker compose ps           # все healthy, migrate — exited (0)

# 6. Засеять справочники объекта (помещения/узлы/камеры) — см. §5
```

**Требования к серверу:** Linux x86-64, Docker + Docker Compose v2, несколько ядер
CPU (GPU не нужен), узлы датчиков и камеры — в одной LAN с сервером.

## 4. Где какие файлы лежат (карта конфигурации заказчика)

Всё, что заказчик настраивает под свой объект, собрано в нескольких местах. **Файлы
с секретами и реальными адресами в репозиторий не коммитятся** (есть `.example`).

| Что настраиваете | Файл | Шаблон в репозитории |
|---|---|---|
| Секреты и параметры окружения | `.env` | `.env.example` |
| Справочник объекта (помещения, узлы, камеры) | `db/seeds/object.yaml` | `db/seeds/object.example.yaml` |
| Камеры для медиа-шлюза (RTSP/ONVIF) | `media-gateway/go2rtc.yaml` | в репозитории |
| Расписания видеоанализа | `config/schedules.json` | `config/schedules.example.json` |
| Модель распознавания поз | `models/pose_landmarker.task` | внешний ассет (`models/README.md`) |
| Прошивки узлов датчиков | свои копии | `firmware/esphome/*.example.yaml` |

Структура всего репозитория описана в [`docs/01_ARCHITECTURE.md`](01_ARCHITECTURE.md) §8.

Артефакты видеоаналитики (скриншоты-доказательства) хранятся в общем томе Docker
`artifacts`; пути к ним приходят в событиях (`artifact_path`).

## 5. Заведение объекта: помещения, датчики, камеры

Помещения, узлы датчиков и камеры — это **справочник объекта**. Он заполняется
из конфига `db/seeds/object.yaml` и загружается в базу скриптом-сидером
(идемпотентно, повторный запуск безопасен).

### 5.1 Заполнить конфиг

```bash
cp db/seeds/object.example.yaml db/seeds/object.yaml
```

Отредактируйте `db/seeds/object.yaml` под объект:

```yaml
rooms:
  - id: room-01                       # короткий идентификатор (латиницей)
    name: "Помещение для приготовления пищи"
    is_cold: false
  - id: room-02
    name: "Холодильная камера"
    is_cold: true

sensor_nodes:
  - id: node-01
    room_id: room-01                  # ссылка на помещение выше
    placement: "внутри (I2C)"
    power: mains

cameras:
  - id: "11111111-1111-1111-1111-111111111111"   # UUID (сгенерировать: uuidgen)
    room_id: room-01
    name: cam-01                       # ВАЖНО: = имя потока в go2rtc.yaml (см. §6)
    rtsp_url: "rtsp://camera.local/stream"
    enabled: true
```

### 5.2 Загрузить в базу

```bash
# Проверка без записи (печатает, что вставит):
DATABASE_URL=postgresql+psycopg2://monitoring:$POSTGRES_PASSWORD@localhost:5432/monitoring \
  python scripts/seed.py db/seeds/object.yaml

# Применить (UPSERT по id — можно повторять при изменениях):
DATABASE_URL=postgresql+psycopg2://monitoring:$POSTGRES_PASSWORD@localhost:5432/monitoring \
  python scripts/seed.py db/seeds/object.yaml --apply
```

После этого камеры появятся в веб-GUI (§7), где можно настроить функции аналитики
и рабочие зоны. Отдельную камеру можно также завести прямо из веб-GUI или по REST
(`POST /api/v1/cameras`, см. §9.3) — сид-конфиг удобен для первоначального
массового заполнения, GUI — для добавления одной камеры на ходу.

## 6. Камеры и медиа-шлюз (go2rtc)

Видеопотоки камер принимает `media-gateway` (go2rtc). Опишите камеры в
[`media-gateway/go2rtc.yaml`](../media-gateway/go2rtc.yaml):

```yaml
streams:
  cam-01: rtsp://camera.local/stream    # имя потока (cam-01) = cameras.name в сиде
```

**Ключевая связка:** имя потока в `go2rtc.yaml` должно совпадать с полем `name`
камеры в `db/seeds/object.yaml`. По этому имени веб-GUI запрашивает кадр-превью
для разметки зон; при несовпадении кнопка «Загрузить кадр» вернёт ошибку.

---

## 7. Веб-интерфейс настройки видеоаналитики

Откройте в браузере: **`http://<сервер>:8000/ui/`**

1. В шапке введите **API-ключ** — это значение `API_KEY` из вашего `.env`
   (см. §9.1). Ключ сохранится в браузере.
2. Нажмите **«Загрузить камеры»** — появится список заведённых камер. Если список
   пуст — разверните **«Добавить камеру»** и заведите камеру (помещение, имя =
   поток в go2rtc, `rtsp://…`), либо засейте справочник (§5).
3. Выберите камеру:
   - флажок **«камера включена»** — общий выключатель аналитики камеры;
   - флажки функций: **позы / действия / спецодежда / покрытие** — что считать;
   - **«Сохранить настройки»**.
4. **Разметка рабочих зон (ROI)** для расчёта % покрытия:
   - **«Загрузить кадр»** — подтянется текущий кадр камеры (через go2rtc);
   - кликами по кадру обведите многоугольник зоны (≥3 вершины);
   - выберите **тип зоны** (стол / пол / окно) и нажмите **«Сохранить зону»**;
   - зоны видны в таблице ниже, есть кнопка удаления.

Изменения учитываются видеоаналитикой при следующем задании. Без зон аналитика
работает, просто не считает покрытие.

> Если список камер пуст — камеры ещё не заведены в справочнике объекта (см. §5).

## 8. Дашборды (Grafana)

Откройте **`http://<сервер>:3000`**, войдите как `admin` с паролем
`GF_SECURITY_ADMIN_PASSWORD` из `.env`. Дашборды (показания датчиков, журнал
событий) преднастроены. Grafana читает базу в режиме «только чтение».

## 9. Работа с API

Машинно-читаемая спецификация всегда доступна у работающего сервера:
**`http://<сервер>:8000/docs`** (Swagger UI) и **`/openapi.json`**. Полный
контракт — [`docs/03_API_CONTRACT.md`](03_API_CONTRACT.md).

### 9.1 Аутентификация

Публичные эндпойнты требуют заголовок **`X-API-Key`** со значением переменной
**`API_KEY`** из `.env` сервера. Этот ключ заказчик назначает сам при
развёртывании (придумать секрет, прописать в `.env`, перезапустить `api-gateway`).
Тот же ключ вводится в шапке веб-GUI. Неверный/отсутствующий ключ → `401`.

### 9.2 Формат ответа (конверт)

Все JSON-ответы обёрнуты в единый конверт:

```json
{ "status": "ok", "data": { "...": "полезная нагрузка" }, "error": null,
  "ts": "2026-06-08T10:30:00Z" }
```

При ошибке: `status:"error"`, `data:null`, `error:{ "code": "...", "message": "..." }`.
Коды ошибок: `VALIDATION_ERROR` (422), `UNAUTHORIZED` (401), `TASK_NOT_FOUND`,
`EVENT_NOT_FOUND`, `CAMERA_NOT_FOUND`, `ZONE_NOT_FOUND` (404), `NOT_IMPLEMENTED`
(501, разъёмы АУРА), `INTERNAL` (500).

### 9.3 Примеры запросов

```bash
KEY="$API_KEY"; H="X-API-Key: $KEY"; BASE="http://localhost:8000/api/v1"

# Здоровье сервиса
curl -H "$H" $BASE/health

# Журнал событий (фильтры: from, to, type, room, limit, offset)
curl -H "$H" "$BASE/events?room=room-02&type=threshold_exceeded&limit=50"

# Одно событие
curl -H "$H" $BASE/events/<event-id>

# Показания датчиков (основной просмотр — Grafana; это для проверки/интеграций)
curl -H "$H" "$BASE/readings?room=room-01&metric=air_temp&from=2026-06-01T00:00:00Z"

# Поставить разовое задание видеоанализа (помимо расписания)
curl -H "$H" -H "Content-Type: application/json" -X POST $BASE/analysis-tasks -d '{
  "source_type": "stream",
  "source_ref": "rtsp://camera.local/stream",
  "room": "room-01",
  "camera_id": "11111111-1111-1111-1111-111111111111",
  "pipeline": "pose_v1",
  "params": { "fps": 5 }
}'
# camera_id (опц.) применит к заданию тумблеры и зоны этой камеры —
# в т.ч. при анализе файла (source_type:"file", source_ref: путь на общем томе)

# Статус/результат задания
curl -H "$H" $BASE/analysis-tasks/<task-id>

# Камеры и зоны (то же, что делает веб-GUI)
curl -H "$H" $BASE/cameras
curl -H "$H" -X PATCH $BASE/cameras/<id> -H "Content-Type: application/json" \
  -d '{ "enabled": true, "analytics": { "coverage": true } }'
curl -H "$H" -X POST $BASE/cameras/<id>/zones -H "Content-Type: application/json" \
  -d '{ "zone_type": "table", "polygon": [[0.1,0.1],[0.5,0.1],[0.5,0.5]] }'
curl -H "$H" -X DELETE $BASE/zones/<zone-id>
```

## 10. События: что система сообщает

Каждое событие имеет `source` (`sensors` | `analytics`), `type`, `message`
(готовая фраза на русском для оператора) и `payload` (машинные детали).

| source | type | смысл | пример сообщения |
|---|---|---|---|
| sensors | `threshold_exceeded` | показание вышло за порог | «В холодильной камере температура выше нормы» |
| sensors | `back_to_normal` | показание вернулось в норму | «Влажность вернулась к норме» |
| sensors | `sensor_silent` | узел молчит дольше допустимого | «Датчик в холодильной камере молчит 12 минут» |
| analytics | `pose_event` | простая поза (рука/колено/корпус) | «Поднята правая рука» |
| analytics | `action_detected` | составное действие с длительностью | «Протирание поверхности двумя руками, 4 с» |
| analytics | `coverage_report` | % покрытия рабочей зоны | «Покрытие зоны стола — 63%» |
| analytics | `condition_flagged` | эвристика спецодежды | «Не распознана спецодежда (белый халат)» |

Подробнее — [`docs/04_DATA_MODEL.md`](04_DATA_MODEL.md) §4 и
[`docs/07_VIDEO_ANALYTICS.md`](07_VIDEO_ANALYTICS.md).

## 11. Расписания видеоанализа

В v1 видеоанализ запускается **по расписанию** планировщиком. Настройка —
`config/schedules.json` (шаблон `config/schedules.example.json`):

```bash
cp config/schedules.example.json config/schedules.json
```

Каждая запись задаёт `source_ref` (поток камеры), `room_id`, `camera_id` (чтобы
считалось покрытие ROI) и `interval_min`. Без файла планировщик работает вхолостую.

## 13. Граница v1 / v2 и интеграция с АУРА

В версии 1 наш контур **самостоятелен**. Разъёмы интеграции с АУРА
(`/api/v1/integration/*`) физически существуют, но **выключены** и возвращают
`501 NOT_IMPLEMENTED`. Их включение в v2 — через фичефлаг
`AURA_INTEGRATION_ENABLED`, без переписывания кода и без смены путей. В v1:

- показания датчиков в АУРА **не передаются** (копятся только у нас);
- наружу контур отдаёт **события**, а не сырые ряды и не видеозапись;
- видеоанализ запускается по нашему расписанию (в v2 — ещё и по команде АУРА,
  в т.ч. по переданному видеофайлу).

Детали стыка — [`docs/03_API_CONTRACT.md`](03_API_CONTRACT.md) §4.

## 14. Типичные вопросы и проблемы

| Симптом | Причина и решение |
|---|---|
| В GUI пустой список камер | Камеры не заведены — добавьте камеру в GUI («Добавить камеру», §7) либо заполните `db/seeds/object.yaml` и выполните сидер (§5). |
| `401 UNAUTHORIZED` на запросах | Неверный `X-API-Key`. Используйте значение `API_KEY` из `.env` сервера (§9.1). |
| «Кадр недоступен» при разметке | Имя камеры (`cameras.name`) не совпадает с именем потока в `go2rtc.yaml`, либо камера недоступна (§6). |
| Сервис `video-analytics` не стартует | Нет файла модели `models/pose_landmarker.task` (§3, `models/README.md`). |
| Нет событий покрытия (`coverage_report`) | Не размечены ROI-зоны камеры — добавьте зоны в GUI (§7). |
| Эндпойнт `/integration/*` отдаёт `501` | Это норма для v1: интеграция с АУРА выключена (§13). |
