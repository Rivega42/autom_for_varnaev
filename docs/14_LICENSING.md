# 14 · Лицензирование (демо-лимиты и тарифы)

> Модель продаж: бесплатное **демо** ограничено (1 помещение, 1 камера, 1 узел
> датчиков); расширение — по **лицензионному ключу** с тарифными лимитами и
> сроком (подписка). Здесь — архитектура: что и как ограничиваем, как устроен
> ключ, и зачем (и когда) нужен сервер лицензий + личный кабинет.

---

## 1. Что ограничиваем

Лимит — на **число активных сущностей справочника**, проверяется при заведении
(`POST /rooms`, `/cameras`, `/sensor-nodes`). Мягко удалённые камеры (`deleted_at`)
не считаются. Аналитика/датчики работают без ограничений по объёму данных —
платим за масштаб объекта, а не за поток.

| Тариф | Помещения | Камеры | Узлы датчиков |
|---|---|---|---|
| **Демо** (без ключа) | 1 | 1 | 1 |
| По ключу | из ключа | из ключа | из ключа (или `unlimited`) |

Превышение → `409 LICENSE_LIMIT` с человекочитаемым сообщением и подсказкой
ввести ключ.

---

## 2. Честно про защиту (важно)

Продукт работает **на сервере клиента** — клиент владеет хостом и имеет root
(см. `docs/05`). Это значит: **ни одна схема не делает лицензию невзламываемой**
для самого владельца сервера — он может пропатчить код/бинарь, подменить env,
заглушить любой сетевой сервер лицензий. Это касается и офлайн-подписи, и
онлайн-сервера лицензий одинаково.

Поэтому цель лицензирования здесь — **не DRM против взломщика**, а:
- барьер для честных пользователей (нельзя случайно «дорисовать» лимит);
- коммерческий и юридический рычаг (имя клиента и срок зашиты в ключ, подделка
  ломает подпись);
- учёт и удобство продления.

С этим пониманием выбираем самую надёжную для on-prem схему, не переусложняя.

---

## 3. Механизм enforcement: офлайн-подписанные ключи (Ed25519)

**Это то, что встраивается в продукт.** Работает без интернета — критично для
объектов с изолированной/нестабильной сетью.

- У вендора — **приватный** ключ Ed25519 (в репозиторий **не** попадает).
- В продукт зашит соответствующий **публичный** ключ (его публичность —
  нормальна; репозиторий открыт).
- Лицензия — строка `payload.signature` (base64url):
  ```json
  payload = {
    "customer": "ООО Пекарня №1",
    "tier": "pro",
    "max_rooms": 10, "max_cameras": 20, "max_nodes": 30,
    "issued": "2026-06-13", "expires": "2027-06-13"
  }
  ```
  `signature` = Ed25519 над байтами `payload`. Продукт проверяет подпись зашитым
  публичным ключом — **офлайн, подделать лимиты/срок нельзя** (сломается подпись).
- **Подписка**: после `expires` — мягкий откат к демо-лимитам. Систему **не
  ломаем**: уже заведённые сущности продолжают работать, блокируется только
  добавление сверх демо; в GUI — предупреждение «лицензия истекла, продлите».
  Продление = выдача нового ключа с новым `expires`.
- **Где ключ**: переменная `LICENSE_KEY` и/или поле в GUI (хранится в БД, таблица config).

Ограничение офлайн-схемы: **нельзя отозвать ключ до истечения** (утёкший ключ
действует до `expires`). Смягчается коротким сроком подписки (ключи на год,
продление ежегодно) — этого достаточно для старта продаж.

## 4. Сервер лицензий и личный кабинет (фаза 2)

Сервер лицензий нужен **не для enforcement** (он остаётся офлайн — см. §2), а для
**операционной части бизнеса**: выпуск, продление, отзыв, самообслуживание,
биллинг. Это **отдельный продукт** (SaaS), который не требуется, чтобы начать
продавать — на старте ключи выпускаются вручную утилитой (`scripts/gen_license`)
по договору.

Когда объём вырастет, поднимается **личный кабинет (сайт)**:

```
┌─────────────────────┐      выпуск/продление/отзыв      ┌──────────────────────┐
│  Личный кабинет     │  ──────────────────────────────▶ │  Сервис лицензий     │
│  (клиент: продлить, │                                   │  (приватный ключ,    │
│   скачать ключ)     │  ◀── ключ (Ed25519-подписанный) ─ │   БД лицензий,       │
│  (вендор: клиенты,  │                                   │   биллинг)           │
│   биллинг, отзыв)   │                                   └──────────────────────┘
└─────────────────────┘                                            │ подписывает тем же
        ▲                                                          │ приватным ключом
        │ скачать ключ                                             ▼
        │                                              ┌──────────────────────────┐
   контур на объекте  ◀── вставка ключа в GUI/env ──── │ строка лицензии (офлайн) │
   (проверяет офлайн)                                  └──────────────────────────┘
```

Что добавляет сервер/кабинет:
- **выпуск и продление** ключей без ручной утилиты (тот же Ed25519-подпись);
- **отзыв** до истечения — реализуется **онлайн-проверкой** как *опциональным*
  слоем: контур, если у объекта есть интернет, периодически «звонит домой» и
  узнаёт отозванные ключи; при недоступности сервера — **не блокирует** (grace),
  чтобы изолированный объект не встал. Это soft-revocation поверх офлайн-основы;
- **самообслуживание** (клиент сам продлевает/скачивает), биллинг, телеметрия
  активаций.

**Вывод по архитектуре:** enforcement в продукте — **офлайн-подпись** (строим
сейчас). Сервер лицензий + личный кабинет — **фаза 2**, отдельный сервис,
переиспользует ту же подпись; на старте заменяется ручным выпуском ключей.

---

## 5. Контракт (что появится в API/GUI)

- `GET /api/v1/license` — текущий тариф, лимиты, расход (использовано/лимит),
  `expires`, статус (`demo` | `active` | `expired` | `invalid`). Для баннера в GUI.
- `POST /rooms` / `/cameras` / `/sensor-nodes` при превышении → `409 LICENSE_LIMIT`.
- GUI: баннер тарифа в шапке («Тариф: демо · камеры 1/1») + поле ввода ключа;
  понятное сообщение при достижении лимита.

---

## 6. Инструменты вендора (не в проде, приватный ключ вне репо)

- `scripts/gen_license_keypair.py` — один раз: сгенерировать пару Ed25519;
  публичный ключ зашивается в продукт, приватный хранится у вендора.
- `scripts/gen_license.py` — выпустить ключ: подписать payload (клиент, тариф,
  лимиты, срок) приватным ключом → строка лицензии. Приватный ключ берётся из
  файла/переменной, **в репозиторий не коммитится**.

---

## 7. План реализации (по PR)

1. **Ядро + enforcement (backend):** модуль проверки лицензии (Ed25519, лимиты,
   срок, демо по умолчанию), зашитый публичный ключ, enforcement в трёх `POST`,
   `GET /license`, утилиты вендора, тесты. Зависимость `cryptography`.
2. **GUI:** баннер тарифа + поле ввода ключа + сообщения о лимите.
3. **(Фаза 2, отдельно)** сервер лицензий + личный кабинет; опциональная
   soft-revocation онлайн-проверкой.

Граница: лицензирование — наш контур; к АУРА отношения не имеет (v1 standalone).
