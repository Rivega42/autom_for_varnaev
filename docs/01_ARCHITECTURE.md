# 01 · Архитектура

## 1. Контекст

Комплекс устанавливается на **объекте**. На объекте несколько **помещений**,
каждое оснащается **датчиками** (температура воздуха, влажность, бесконтактная
ИК-температура поверхности) и **камерами**. Всё сводится на **один физический
сервер на объекте**.

На сервере в контейнерах сосуществуют два продукта:

- **Наш контур** — приём и накопление данных датчиков, видеоаналитика, единый
  журнал событий. **Это то, что мы разрабатываем.**
- **АУРА** — продукт заказчика: контроль производства, тех. карты, программа
  производственного контроля (ППК), видеозапись. **Мы её не разрабатываем.**

В **версии 1** наш контур — **standalone**: не вызывает АУРА и не вызывается ею,
но архитектурно подготовлен к будущему стыку.

## 2. Верхнеуровневая схема

См. `docs/diagrams/01_topology.svg`.

```
ОБЪЕКТ
 ├─ Помещение 1 ─ датчики + камера(ы)
 ├─ Помещение 2 ─ датчики + камера(ы)
 └─ Помещение N ─ датчики + камера(ы)
        │ датчики → MQTT          │ камеры → RTSP
        ▼                         ▼
 ┌──────────────────────────────────────────────────────────────┐
 │  ФИЗИЧЕСКИЙ СЕРВЕР (Docker)                                    │
 │                                                                │
 │  [НАШ КОНТУР — контейнеры]                                     │
 │    mqtt-broker → ingest-sensors → ┐                            │
 │                                   ├→ TimescaleDB ← grafana     │
 │    камеры →RTSP→ media-gateway → video-analytics ─┘   ▲        │
 │         │ события + артефакты (скриншот + keypoints)  │        │
 │         └──────────────→ log-service ────────────────┘        │
 │    scheduler (триггер аналитики по расписанию)                 │
 │    api-gateway  ←── единственный вход снаружи контура          │
 │                                                                │
 │  [АУРА — контейнеры заказчика] (мы не трогаем)                 │
 │    видеозапись · контроль производства · тех.карты · ППК       │
 │                                                                │
 │  [общий том артефактов] ← пишем мы, читает АУРА (в v2)         │
 └──────────────────────────────────────────────────────────────┘
```

## 3. Компоненты нашего контура

| Контейнер | Назначение | Стек |
|---|---|---|
| `mqtt-broker` | приём данных датчиков по MQTT | Mosquitto |
| `ingest-sensors` | подписка на MQTT, парсинг, запись показаний в БД, контроль «тишины» | Python/FastAPI (воркер) |
| `media-gateway` | приём RTSP/ONVIF с камер, релей потоков воркеру аналитики и превью в браузер (WebRTC), снимает проблему CORS дешёвых камер | go2rtc (альт. MediaMTX) |
| `video-analytics` | анализ видео (поток или файл), генерация событий, скриншотов и keypoints/coverage. **Перенос рабочего PoC** (см. `07_VIDEO_ANALYTICS.md`) | Python + MediaPipe (`PoseLandmarker`) |
| `scheduler` | запуск заданий на анализ по расписанию (v1) | Python |
| `log-service` | единый журнал событий: приём от ingest и analytics, чтение через API | Python/FastAPI |
| `api-gateway` | единственный REST-вход контура наружу; разъёмы для АУРА (заглушены в v1) | Python/FastAPI |
| `db` | TimescaleDB: временные ряды датчиков + события/задания/артефакты | TimescaleDB (PostgreSQL 16) |
| `migrate` | одноразовое применение миграций (`alembic upgrade head`) до старта сервисов | Python + Alembic |
| `grafana` | дашборды: графики датчиков, лента событий | Grafana |

Что НЕ наше (контейнеры АУРА, мы их не создаём): видеозапись, контроль
производства, тех. карты, ППК.

## 4. Потоки данных по версиям

### 4.1 Данные датчиков (одинаково в v1 и v2)

```
ESP32-C3 (ESPHome) ──MQTT──► mqtt-broker ──► ingest-sensors ──► TimescaleDB
                                                  └──► событие порога ──► log-service
```

Узлы датчиков — **отдельные контроллеры**, сервер их не опрашивает, они сами
шлют. Состав узла и радиовыбор зафиксированы в решении по датчикам (файл проекта
«Мониторинг_среды_решение.md»): **ESP32-C3 (SuperMini) + SHT41/SHT40** (воздух,
t°/влажность) **+ MLX90614 / GY-906** (бесконтактная ИК-температура поверхности)
на одной шине I²C, прошивка **ESPHome** (YAML, без кода). Радио v1 — **Wi-Fi**;
Zigbee/LoRa — точечно для «трудных» точек. Холодильные камеры: контроллер с радио
и питанием **снаружи**, датчики внутри на I²C-шлейфе через уплотнитель двери
(камера — клетка Фарадея, радио сквозь дверь не проходит).

### 4.2 Видео — **меняется** между версиями

**v1:** поток идёт двум независимым потребителям. RTSP с камер заводится в наш
**media-gateway (go2rtc)**, который релеит поток воркеру аналитики и отдаёт превью
в браузер по WebRTC (заодно снимает проблему CORS дешёвых камер — у них нет
`Access-Control-Allow-Origin`, из-за чего прямое чтение кадров «портит» канвас).

```
камера ─RTSP─┬─► media-gateway (go2rtc) ─► video-analytics (наш, анализ по расписанию)
             └─► АУРА (видеозапись, своим трактом)
```

**v2:** владение потоком уходит в АУРА, нам прилетают фрагменты на анализ.

```
камера ─RTSP─► АУРА (запись + владение потоком)
                  └─► REST к нам: "проанализируй вот этот файл/фрагмент"
                       (путь к файлу на общем томе) ─► video-analytics
```

**Ключевое архитектурное следствие:** `video-analytics` уже в v1 принимает
**источник как параметр** — `stream` (RTSP/WebRTC через media-gateway) ИЛИ `file`
(путь на томе). В v1 используется только `stream`, режим `file` реализован, но
дёргается лишь нашим планировщиком/тестами. В v2 «файл от АУРА» — это тот же режим
`file`, просто триггер приходит по REST. Никакого переписывания движка.

### 4.3 Триггер аналитики — **расширяется** между версиями

```
v1:  scheduler (cron) ──► создаёт "задание на анализ" ──► video-analytics
v2:  scheduler (cron) ──┐
     АУРА (REST)     ───┴► создаёт "задание на анализ" ──► video-analytics
```

«Задание на анализ» (`analysis_task`) — сущность первого класса уже в v1.
Планировщик v1 и интеграция v2 — это **один механизм с двумя источниками
триггера**. Поэтому задание заводим как полноценную запись в БД со статусами
(см. `docs/04_DATA_MODEL.md`).

## 5. Единый журнал событий

Точка схождения сенсорики и видеоаналитики. **Обе подсистемы наружу отдают
события, а не сырьё** — это общий принцип контура:

- видеоаналитика выдаёт распознанные **события** (поза, действие, покрытие), а не
  сырые кадры;
- подсистема датчиков выдаёт **события**, выведенные из показаний **по критериям**
  (пороги, «тишина» узла), а не сами временные ряды.

В журнал пишут:

- `ingest-sensors` — события датчиков (превышение порога t°/влажности, возврат к
  норме, «датчик молчит дольше X» — критично для холодовой цепи);
- `video-analytics` — события аналитики (например, распознанное действие; кадр-
  доказательство сохраняется как **артефакт**, в событие кладётся ссылка на него).

Каждое событие несёт **человекочитаемое сообщение** для оператора
(`events.message`, по-русски, с контекстом помещения) — например «В холодильной
камере температура выше нормы» или «В помещении для приготовления пищи влажность
вернулась к норме». Машинные детали остаются в `payload`.

**Граница АУРА.** В v1 ни показания датчиков, ни события в АУРА не передаются
(интеграция заглушена); показания копятся в нашей БД и доступны нам (Grafana,
`GET /readings`). Журнал — будущая точка интеграции: в v2 АУРА сможет читать
**события** (датчиков и аналитики) через `api-gateway`, но не сырые ряды показаний.
Схема события — `docs/04_DATA_MODEL.md`.

## 6. Артефакты (скриншоты сейчас, видеофайлы потом)

Скриншот-доказательство, который мы пишем сами (v1), и видеофайл, который
пришлёт АУРА (v2), — это **один абстрактный артефакт с путём**. Храним по
предсказуемой схеме на **общем томе**:

```
/data/artifacts/<YYYY-MM-DD>/<event_id|task_id>.<ext>
```

В журнал/задание кладём **ссылку** (путь), а не сам файл. По REST между нами и
АУРА ходят только метаданные и пути — бинарь идёт через общий том. В v2 «файл от
АУРА» — просто ещё один источник артефактов, не новая подсистема.

**Права общего тома (важно для эксплуатации).** Прикладные сервисы
(`video-analytics`, `api-gateway`) работают под непривилегированным пользователем
`appuser` (uid 10001), а Docker инициализирует named-volume `artifacts` владельцем
`root` — то есть «из коробки» `appuser` не может создать подкаталог по дате и
сохранить кадр-улику (`PermissionError`). Поэтому в `docker-compose` есть
one-shot-сервис **`artifacts-init`** (по образцу `migrate`): до старта сервисов он
выставляет владельца тома `10001:10001`. Это декларативно, переживает пересоздание
тома и входит в поставку (release-compose + офлайн-бандл) — ручной `chmod` на томе
не требуется. Все писатели тома используют один uid (10001) — в том числе
release-сборка `api-gateway`, — поэтому подкаталоги по дате не «разъезжаются» по
владельцам.

## 7. Граница версий (сводка)

| Аспект | v1 | v2 |
|---|---|---|
| Наш контур | рабочий, standalone | + интеграция с АУРА |
| Видеопоток | к нам и в АУРА раздельно | в АУРА, к нам — фрагменты |
| Источник для аналитики | `stream` | `stream` и `file` |
| Триггер аналитики | расписание | расписание + REST от АУРА |
| `/integration/*` REST | есть, **заглушены** (501 / фичефлаг) | включены |
| Сеть `integration` | создана, обмена нет | обмен с АУРА |

## 8. Структура репозитория

Фактическая структура монорепо (v1 реализован). Следовать ей, не придумывать свою.

```
.
├─ CLAUDE.md
├─ README.md
├─ docs/                      # эта документация (источник истины)
├─ docker-compose.yml         # все контейнеры, две сети, общий том
├─ .env.example               # пример переменных (без секретов)
├─ .gitignore
├─ services/
│  ├─ api-gateway/            # FastAPI: внешний REST-вход + разъёмы АУРА (заглушены)
│  │  ├─ app/
│  │  ├─ tests/
│  │  ├─ Dockerfile
│  │  └─ pyproject.toml
│  ├─ ingest-sensors/         # MQTT → БД, контроль "тишины"
│  │  ├─ app/  ├─ tests/  ├─ Dockerfile  └─ pyproject.toml
│  ├─ video-analytics/        # MediaPipe PoseLandmarker; источник = stream|file
│  │  ├─ app/                 # порт движка из PoC (события, действия, зоны, покрытие)
│  │  ├─ reference/           # ИСХОДНЫЙ PoC motion-log.html — эталон поведения для порта
│  │  ├─ tests/  ├─ Dockerfile  └─ pyproject.toml
│  ├─ scheduler/              # триггер заданий по расписанию (v1)
│  │  ├─ app/  ├─ tests/  ├─ Dockerfile  └─ pyproject.toml
│  └─ log-service/            # единый журнал событий
│     ├─ app/  ├─ tests/  ├─ Dockerfile  └─ pyproject.toml
├─ media-gateway/             # go2rtc.yaml: приём RTSP/ONVIF, релей воркеру + превью WebRTC
├─ db/
│  ├─ migrations/             # Alembic
│  ├─ init/                   # включение timescaledb, ro-пользователь Grafana
│  └─ Dockerfile              # образ сервиса migrate (alembic upgrade head)
├─ firmware/
│  └─ esphome/                # YAML-конфиги узлов (ESP32-C3 + SHT4x + MLX90614)
├─ grafana/
│  ├─ dashboards/             # дашборды (sensors, events)
│  └─ provisioning/           # datasource (ro), дашборды, алерты
├─ models/                    # бинарная модель MediaPipe (внешний ассет, не в гите)
├─ config/                    # расписания планировщика (schedules.json)
├─ shared/                    # общие Python-модели (конверт API, схемы событий)
├─ tests/                     # корневые тесты (структура, миграции, compose)
└─ scripts/                   # утилиты разработчика (проверки, сиды)
```

`shared/` важен: единый конверт API и схемы событий/заданий лежат там, чтобы
сервисы не расходились в форматах.

> **Имя Python-пакета сервиса.** В дереве выше папка кода показана как `app/`
> условно. Фактически каждый сервис экспортирует **уникально именованный**
> пакет (напр. `ingest_sensors`, `log_service`, `api_gateway`), а не общий `app`,
> — иначе одноимённые пакеты разных сервисов сталкиваются при общем прогоне
> тестов в монорепо.

## 9. Обоснование стека

- **Python 3.12 + FastAPI везде.** Один язык для всей команды («своими силами») и
  для Кода — меньше переключений контекста. FastAPI даёт REST + валидацию
  (Pydantic) + автогенерацию OpenAPI (`/docs`, `/openapi.json`) — это и есть
  машинно-читаемая версия контракта.
- **MediaPipe `PoseLandmarker` для аналитики — переносим рабочий PoC, не пишем
  заново.** Движок уже собран и проверен в браузерном PoC (`motion-log.html`): 33
  точки тела, простые события и составные действия, ROI-зоны и % покрытия,
  цветовая эвристика «белого халата». У MediaPipe есть Python-пакет (Tasks) —
  переносим тот же `PoseLandmarker` и ту же логику в headless-воркер. Полная
  спецификация поведения — `docs/07_VIDEO_ANALYTICS.md`. Нагрузка: на малом числе
  камер при пониженном fps (5–8) достаточно **CPU**; GPU-батчинг — рычаг на случай
  роста числа камер, в v1 не требуется.
- **media-gateway (go2rtc).** RTSP/ONVIF-камеры заводим через релей: он отдаёт
  поток воркеру и превью в браузер (WebRTC) и снимает CORS-проблему дешёвых камер.
  Знаком по прошлым работам (MediaMTX/WHEP); go2rtc легче, MediaMTX — альтернатива.
- **TimescaleDB вместо InfluxDB + отдельной реляционной БД.** Одна база покрывает
  и временные ряды (показания датчиков — hypertable), и реляционные сущности
  (события, задания, артефакты). Это знакомый PostgreSQL, отлично поддержан
  инструментами и Кодом. Меньше движущихся частей.
- **Mosquitto (MQTT).** Стандарт для телеметрии, ESPHome шлёт в него нативно;
  лёгкий, без облака.
- **Grafana.** Готовые дашборды и алертинг поверх Postgres/Timescale — минимум
  собственного UI-кода в v1.
- **Docker Compose.** Один хост, несколько контейнеров — это ровно сценарий
  Compose. Сегментация сетей и общий том описываются декларативно (см. `02`).

Замечание про защиту кода: Python в образе читаем. Сервисы, которые мы хотим
закрыть, на этапе упаковки компилируются (Nuitka) и кладутся в distroless. Это
не меняет язык разработки. Подробно — `docs/05_SECURITY_CODE_PROTECTION.md`.
