# 05 · Защита кода в контейнере

Документ отвечает на вопрос: «можно ли закрыть наш контейнер так, чтобы не было
доступа к коду?» — честно, без иллюзий, с практическими мерами.

---

## 1. Короткий честный ответ

**Полностью закрыть код от того, кто владеет хостом, — нельзя.** Сервер стоит у
заказчика, у заказчика есть `root`, а `root` на хосте может всё:

- `docker save <image>` → распаковать слои → прочитать файлы;
- `docker cp` / прямой доступ к слоям в `/var/lib/docker`;
- зайти в работающий контейнер и прочитать ФС;
- снять дамп памяти процесса.

**Контейнер — это изоляция выполнения, а не защита исходников.** Любая защита на
хосте, которым владеет другая сторона, — это «поднять планку», а не «запретить».

Поэтому стратегия двухслойная: **технически поднимаем планку** (чтобы случайно
заглянуть и чтобы рядовому разработчику было трудно) + **юридически закрываем**
(лицензия/NDA — в B2B это часто и есть основной работающий механизм).

---

## 2. Что нам мешает и что помогает

- **Python — читаемый по умолчанию.** Если положить `.py` в образ, исходник
  открыт. Даже `.pyc` декомпилируется. Значит для закрываемых сервисов нужен шаг
  компиляции/упаковки.
- **v1 — оффлайн, standalone.** Значит лицензионный сервер / «phone-home» /
  удалённая активация **не работают** (нет связи). На v1 защита — только
  локальная (упаковка) + договор.

---

## 3. Меры по уровням (от обязательного к усиленному)

### Уровень 0 — гигиена (делаем всегда, бесплатно)
- **Multi-stage build:** исходники только в build-стадии; в финальный образ
  попадает лишь артефакт. Никаких `.py`, `.git`, тестов, dev-зависимостей в
  runtime-образе.
- **`.dockerignore`** исключает `.git`, тесты, доки, `.env`.
- **Никаких секретов в образе** — только через переменные окружения в рантайме.

### Уровень 1 — distroless / минимальный runtime
- Базовый образ **distroless** или `scratch`: нет шелла (`sh`), нет пакетного
  менеджера, минимум утилит. Залезть внутрь и «походить по ФС» становится сильно
  труднее. Поверхность атаки минимальна.

### Уровень 2 — компиляция Python (основная мера для закрываемых сервисов)
- **Nuitka** — компилирует Python в нативный бинарь (C под капотом). В образ
  кладётся бинарь, не исходник. Это не делает реверс невозможным, но превращает
  «открыл и прочитал» в «дизассемблируй и разбирайся» — другой класс усилий.
- Альтернатива/дополнение — **Cython** для критических модулей.
- **Обфускация (PyArmor)** — слабее компиляции (по сути шифрование байткода с
  загрузчиком), но дёшева; подходит как дополнительный слой.

### Уровень 3 — сменить язык для самого ценного (если оправдано)
- Сервис с «секретным соусом» можно написать на **Go/Rust** → один статический
  бинарь, исходников в образе нет вовсе, distroless/scratch-runtime. Реверс
  возможен, но это уже целенаправленная работа, а не чтение.
- Цена: второй язык в проекте (сложнее для команды «своими силами» и для Кода).
  Поэтому — только для действительно ценного модуля, не для всего контура.

### Уровень 4 — юридический контур (главное, когда техника бессильна)
- **Лицензионное соглашение** на ПО (запрет декомпиляции/реверса/копирования).
- **NDA** с заказчиком.
- Это реальный механизм защиты в ситуации «ПО на чужом сервере»: он не мешает
  технически прочитать код, но делает использование прочитанного незаконным.

---

## 4. Рекомендация для этого проекта

1. **Везде** — Уровень 0 + Уровень 1 (multi-stage + distroless). Дёшево, и так
   надо делать.
2. **Закрываемые сервисы** (там, где наша логика, прежде всего `video-analytics`
   и при желании `api-gateway`) — **Уровень 2 (Nuitka)** в distroless.
3. **Уровень 3 (Go/Rust)** — держим как опцию, если заказчик потребует более
   серьёзной защиты конкретного модуля; вводим точечно, отдельным решением.
4. **Уровень 4 (лицензия/NDA)** — обязательно, это зона юристов заказчика и наша;
   технически она ничего не закрывает, но закрывает юридически.

**Честно зафиксировать заказчику:** на сервере, которым он владеет, ни один из
этих приёмов не остановит целенаправленный реверс с его стороны. Они
останавливают «случайно посмотрел» и рядового стороннего разработчика. Гарантию
даёт договор, а не контейнер.

---

## 5. Что из этого — задачи Кода (эпик E9)

- Multi-stage Dockerfile для каждого сервиса; runtime — distroless/slim.
- `.dockerignore` во всех сервисах.
- Для `video-analytics` (и опц. `api-gateway`) — стадия сборки через **Nuitka**,
  в runtime только скомпилированный артефакт.
- Проверка: в финальном образе нет `.py`-исходников нашей логики и нет шелла
  (тест на состав образа).
- README-заметка для заказчика с разделом §1 и §4 (пределы + юридический контур).

> Важно: компиляция/обфускация усложняет отладку и сборку. На время активной
> разработки сервисы собираются обычным образом; «закрытая» сборка — отдельный
> профиль (`Dockerfile.release` или build-arg), а не дефолт.
