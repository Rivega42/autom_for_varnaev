# 02 · Сетевая модель

Документ описывает, как контейнеры соединены внутри одного физического хоста, и
**почему именно так** (раздел с обоснованием можно показывать заказчику).

Диаграмма: `docs/diagrams/02_network.svg`.

---

## 1. Решение в одном абзаце

Не плоская сеть. **Сегментация на две Docker-сети + общий том для файлов.**
Наши сервисы общаются между собой по внутренней сети `internal` (наружу не
видны). Наружу из нашего контура смотрит **только** `api-gateway`, подключённый
ко второй сети `integration`, общей с АУРА. Обмен файлами — через общий
**named volume**, а не по HTTP.

---

---

## 3. Кто в какой сети

Сеть `internal` объявлена с `internal: true` — это **реальная** изоляция Docker:
из неё нет выхода наружу, и published-порты контейнеров, висящих только в ней,
не работают. Поэтому есть третья служебная сеть **`edge`** (обычный bridge):
в неё дополнительно включены контейнеры, которым нужен LAN объекта (камеры,
узлы датчиков) или публикация порта на хост. Внутреннее общение сервисов идёт
по-прежнему по `internal`.

| Контейнер | `internal` | `edge` | `integration` | Порт наружу хоста |
|---|:---:|:---:|:---:|---|
| `db` (TimescaleDB) | ✅ | — | — | нет |
| `migrate` (one-shot, миграции) | ✅ | — | — | нет |
| `mqtt-broker` | ✅ | ✅ | — | да (1883: узлы датчиков шлют с LAN объекта*) |
| `media-gateway` (go2rtc) | ✅ | ✅ | — | нет (edge — для доступа К камерам в LAN) |
| `ingest-sensors` | ✅ | — | — | нет |
| `video-analytics` | ✅ | — | — | нет |
| `scheduler` | ✅ | — | — | нет |
| `log-service` | ✅ | ✅ | — | нет (edge — исходящие уведомления Telegram/SMTP/webhook) |
| `grafana` | ✅ | ✅ | — | да (дашборд) |
| `api-gateway` | ✅ | ✅ | ✅ | да (REST) |
| контейнеры АУРА | — | — | ✅ | по усмотрению АУРА |

\* MQTT: узлы датчиков шлют по Wi-Fi из помещений, поэтому брокеру нужен доступ с
физической сети объекта. Это публикация порта брокера на хост-интерфейс LAN, а не
во внешний интернет. Детали — issue в эпике E2.

## 4. Обоснование (для заказчика)

**Изоляция / минимум привилегий.** АУРА видит только REST-эндпойнт
`api-gateway`, но не нашу БД, брокер, лог-сервис и аналитику. Если контейнер
АУРА будет скомпрометирован, он не дотянется до данных напрямую — только до
публичного API, у которого свои проверки. Плоская сеть (все в одной) такой
границы не даёт.

**Нет коллизий портов.** Два продукта на одном хосте, у каждого свои сервисы.
Bridge-сети с изоляцией пространств имён снимают конфликты портов автоматически.
Режим `network_mode: host` мы **не** используем — он сваливает все порты в общую
кучу и ломает изоляцию.

**Переносимость = страховка контракта.** Связь с АУРА идёт по REST поверх TCP, а
не через хитрый IPC/shared-memory. Если заказчик завтра вынесет АУРА на отдельную
машину — наш контракт не меняется ни на строчку, меняется только адрес. Это и
есть «рядом физически, но развязаны логически» (co-located, loosely coupled).

**v1 → v2 без переписывания.** Сеть `integration` существует уже в v1, но
REST-методы для АУРА заглушены (501 / фичефлаг). Включение интеграции в v2 — это
не новая архитектура, а активация уже стоящих разъёмов.

**Файлы — через том, не через API.** Гонять видео и кадры по HTTP дорого и
бессмысленно, когда обе стороны на одном хосте. Общий том: сторона-источник
кладёт файл → по REST передаётся только путь → сторона-получатель читает с тома.

**Цена решения сейчас — почти ноль.** Разделение на две сети в Compose — это
несколько строк. Свести две сети в одну потом легко; разделить уже сросшиеся
продукты потом — больно. Поэтому границу фиксируем сразу.

---

## 5. Контраргумент и ответ

> «v1 же standalone — зачем две сети, поставьте всё в одну.»

Сегментация ничего не стоит сейчас и фиксирует границу безопасности и контракт
между двумя продуктами с самого начала. Объединить сети — обратимая операция в
пару строк; «расцепить» сросшиеся через общую сеть продукты — дорогая. Делаем
дёшево сейчас, чтобы не платить дорого потом.

---

## 6. Что из этого — задачи Кода (эпик E0/E6)

- `docker-compose.yml`: объявить сети `internal` (`internal: true`) и
  `integration`; подключить контейнеры по таблице §3.
- `api-gateway` — единственный в обеих сетях; опубликовать только его и Grafana.
- Объявить named volume `artifacts`, смонтировать в `video-analytics` (rw),
  `api-gateway` (ro). Точка монтирования для АУРА — задокументировать (v2).
- Эндпойнты `/integration/*` — заглушить (см. `docs/03_API_CONTRACT.md`).
