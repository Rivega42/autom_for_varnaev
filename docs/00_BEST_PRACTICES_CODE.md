# 00 · Лучшая практика работы с Claude Code на этом проекте

Документ описывает, **как** мы ведём разработку с Claude Code, чтобы результат был
предсказуемым, проверяемым и удобным для команды. Правила здесь — не догма, но
отклоняться от них стоит осознанно.

---

## 1. Базовый принцип: документы — источник истины

Claude Code хорошо работает, когда у него есть **письменный, однозначный
контекст**. Поэтому:

- Любое архитектурное решение живёт в `docs/`, а не только в голове.
- Если контракт (REST, БД, событие) меняется — сначала меняется документ, потом
  код. В одном и том же PR.
- `CLAUDE.md` — это «долговременная память» Кода. Всё, что должно действовать
  всегда, кладётся туда, а не повторяется в каждой задаче.

---

## 2. Планируй до того, как писать код

Рекомендованный ритм на каждый эпик:

1. **План.** Попроси Код сначала предложить разбивку эпика на атомарные задачи
   (в режиме планирования, без написания кода). Просмотри список.
2. **Issue.** По утверждённой разбивке заводятся issue — по одному на задачу.
3. **Код.** Только после этого Код берёт первый issue и пишет.

Почему: дешевле поправить план из 12 строк, чем переписывать 12 файлов.

---

## 3. Атомарность задач — главное правило дробления

**Атомарная задача** = одно изменение, которое:

- можно описать одним предложением «сделать X, чтобы стало Y»;
- делается за обозримое время;
- проверяется независимо (есть понятный критерий приёмки);
- не тянет «заодно» соседнюю работу.

Признаки, что задача **не** атомарна и её надо дробить:
- в названии есть «и» между разными действиями («сделать ingest **и** дашборд»);
- критерий приёмки — это чеклист на 10 пунктов;
- задача затрагивает несколько сервисов сразу.

**Анти-пример:** «Реализовать контур датчиков».
**Правильно:** разбить на → «поднять Mosquitto в compose» · «создать таблицу
`sensor_readings` (hypertable)» · «ingest: подписка на MQTT-топик» · «ingest:
парсинг payload ESPHome в модель» · «ingest: запись показания в БД» · «ingest:
тест парсинга» · ... — каждое отдельным issue.

---

## 4. Один issue → один PR → один merge

- Каждый issue закрывается **отдельным** Pull Request.
- PR ссылается на issue: в описании `Closes #N`.
- Нашёл по ходу ещё работу — **не делай в этом PR**, заведи новый issue.
- Маленькие PR ревьюятся быстро и редко ломают чужое. Большой PR — это риск.

---

## 5. Эпики, issue, метки, milestone

- **Эпик** = milestone в трекере (или метка `epic:E2`). Объединяет связанные issue.
- **Issue** = одна атомарная задача. Формат — см. `docs/06_BUILD_PLAN.md`.
- **Метки** (labels), единая схема:
  - `epic:E0`…`epic:E9` — принадлежность эпику;
  - `type:feat` / `type:fix` / `type:docs` / `type:infra` / `type:test`;
  - `area:sensors` / `area:analytics` / `area:log` / `area:api` / `area:db` /
    `area:infra` / `area:docs`;
  - `prio:high` / `prio:med` / `prio:low`;
  - `stub-aura` — задача создаёт заглушённый разъём для будущей АУРА (v2).
- Каждый issue имеет: один `epic:*`, один `type:*`, один-два `area:*`, один `prio:*`.

---

## 6. Definition of Done

Issue закрыт, когда выполнены все пункты §8 из `CLAUDE.md`. Кратко: код +
комментарии на русском, тест проходит, линт/типы чистые, документ обновлён (если
менялся контракт), PR ссылается на issue.

---

## 7. Тесты и проверки

- На каждую задачу с логикой — pytest-тест(ы). Нет теста — не закрыто.
- CI прогоняет `ruff`, `mypy`, `pytest` на каждый PR (issue по настройке CI — в эпике E0).
- Контрактные вещи (REST, схемы) тестируются по документу-источнику истины.

---

## 8. Коммиты и ветки

- Ветка под issue: `feat/123-mqtt-ingest`, `docs/45-api-contract`.
- Коммиты по Conventional Commits: `feat: подписка ingest на MQTT-топик\n\nCloses #123`.
- Заголовок коммита короткий (≤72 симв.), тело по-русски поясняет «зачем».

---

## 9. Работа с границей АУРА (частая ошибка)

Код по своей инициативе склонен «доделать до конца» — то есть реализовать
интеграцию с АУРА целиком. **Нам это в v1 не нужно.** Поэтому:

- В каждом таком месте — комментарий `# СТЫК-АУРА (v2): ...`.
- Эндпойнты `/integration/*` существуют, но возвращают `501` либо закрыты
  фичефлагом `aura_integration_enabled=false`.
- Соответствующие issue помечаются меткой `stub-aura`.
- Если Код предлагает «давайте уже сделаем интеграцию» — отклоняем, это v2.

---

## 10. Когда просить Код остановиться и спросить

- Если задача требует решения, которого нет в документах (выбор библиотеки,
  изменение схемы данных) — Код заводит вопрос в issue, а не решает молча.
- Если задача противоречит `CLAUDE.md` — приоритет у `CLAUDE.md`, противоречие
  выносится в комментарий.
- Любое изменение стека из §6 `CLAUDE.md` — только через отдельный issue с
  обоснованием и явным согласием.
