# Research · Mesh для трудных радиозон (Zigbee/Thread/ESP-NOW)

> Статус: **research**, приоритет низкий (issue #209). Открывать, только когда на
> объекте появятся точки, реально недоступные по Wi-Fi. Решение по стеку — всегда
> отдельным issue с обоснованием (CLAUDE.md §6). v1 стек (ESPHome + Wi-Fi + MQTT)
> по умолчанию **не меняем**.

## Вопрос с эксплуатации

Могут ли узлы ESP32 **ретранслировать** данные друг другу:
`датчик → ближайший узел → … → шлюз → Wi-Fi → сервер`? Это нужно, когда узел
стоит там, где Wi-Fi не добивает (толстые стены, холодильные камеры с
металлом, удалённые помещения).

## Что ESP32 умеет аппаратно (факты)

- **ESP-NOW** — лёгкий протокол точка-точка/звезда поверх Wi-Fi-радио, без AP.
  Малое энергопотребление, но **штатно не маршрутизирует многохоп** (нужен
  узел-шлюз; многохоп — руками).
- **ESP-WIFI-MESH (ESP-MDF)** — настоящий Wi-Fi-mesh от Espressif (само-
  организующееся дерево). Тяжёлый, на ESP-IDF.
- **Zigbee / Thread** — на чипах **ESP32-C6 / ESP32-H2** (radio 802.15.4):
  стандартный mesh, где маршрутизирующие узлы (routers) ретранслируют трафик.

## Главное ограничение: ESPHome + mesh

Наш v1 — **ESPHome + Wi-Fi + MQTT** (CLAUDE.md §6). **ESPHome не поддерживает
mesh штатно:**
- ESP-WIFI-MESH — нет;
- ESP-NOW — только **сторонние** внешние компоненты, не из коробки;
- painlessMesh — это Arduino-библиотека, не ESPHome.

Значит любой mesh-путь = либо **уход от ESPHome** на части узлов (своя
прошивка/другой стек), либо **сетевое** решение, не трогающее прошивку.

## Варианты (от дешёвого к «честному mesh»)

### Вариант A. Сетевой уровень — Wi-Fi-репитер / второй AP / Wi-Fi-mesh-роутеры
**Прошивки и backend не трогаем вообще.** Узлы как были подключаются по Wi-Fi;
покрытие расширяем железом сети: репитер, вторая точка доступа в зоне, или
бытовой Wi-Fi-mesh (роутеры с бесшовным роумингом).
- **Плюсы:** ноль изменений в нашем контуре; быстро; дёшево; формат событий
  `monitoring/<node>/<metric>` не меняется.
- **Минусы:** не решает совсем «радио-глухие» точки (металл/экранирование);
  репитер делит полосу.
- **Когда:** первый и основной выбор для большинства «трудных» точек.

### Вариант B. Zigbee/Thread + координатор → Zigbee2MQTT → наш MQTT
Честный стандартный mesh: датчики Zigbee (или узлы на ESP32-C6/H2 с Zigbee-
прошивкой) образуют сеть; маршрутизирующие узлы ретранслируют; **координатор**
(USB-донгл/Ethernet-координатор) собирает данные; **Zigbee2MQTT** публикует их в
**наш же MQTT-брокер**.
- **Плюсы:** настоящий многохоп по стандарту; огромная экосистема готовых Zigbee-
  датчиков; питание от батареи на годы для router-less конечных устройств.
- **Минусы / что прорабатывать:**
  - Наши датчики (SHT41/SHT40, MLX90614) — **не Zigbee**. Либо берём **готовые
    Zigbee-датчики** (но MLX90614-ИК в Zigbee почти нет — придётся ESP32-H2/C6 со
    своей Zigbee-прошивкой, а это **уход от ESPHome** и больше работы), либо
    остаёмся на Wi-Fi-узлах, а Zigbee — только для среды (t/влажность).
  - **Формат топиков.** Zigbee2MQTT публикует в `zigbee2mqtt/<device>/...` с
    другим JSON. Наш `ingest-sensors` ждёт `monitoring/<node>/<metric>` с
    `{"value","unit"}` (см. `docs/08_MQTT_CONTRACT.md`). Нужен **маппинг**:
    либо настройка Z2M (переопределение топика/`homeassistant`-режим + правила),
    либо **тонкий мост** `zigbee2mqtt/* → monitoring/<node>/<metric>` (отдельный
    маленький сервис/правило). Это изменение **на стыке MQTT**, бэкенд-логику
    `ingest-sensors` менять не обязательно.
  - Координатор — единая точка отказа; ставить ближе к центру сети.
- **Стоимость (ориентир):** координатор ~₽2–4 тыс., Zigbee-датчики среды дёшевы;
  ESP32-H2/C6 — дёшевы, но дороже по труду (прошивка).

### Вариант C. ESP-NOW-шлюз (узел-ретранслятор → MQTT)
Конечные узлы шлют по **ESP-NOW** на узел-**шлюз** (он в Wi-Fi), шлюз публикует в
MQTT. Дёшево, низкое энергопотребление на листьях, без Wi-Fi на дальних узлах.
- **Плюсы:** дёшево, только наши ESP32; хорошо для «звезды» вокруг одного шлюза.
- **Минусы:** **уход от ESPHome** (ESP-NOW — своя прошивка/внешние компоненты);
  многохоп — руками; шлюз — единая точка отказа; свой формат → нужен тот же
  маппинг в `monitoring/<node>/<metric>` на шлюзе.
- **Когда:** если Zigbee избыточен, а точек немного и они «вокруг» одного шлюза.

## Сквозные требования (для любого варианта)

- **Формат событий не меняется наружу:** на вход `ingest-sensors` всё должно
  прийти как `monitoring/<node>/<metric>` `{"value","unit"[, "ts"]}`
  (`docs/08_MQTT_CONTRACT.md`). Любой mesh-путь обязан закончиться этим топиком —
  через настройку моста/Z2M, а не правкой контракта.
- **Безопасность:** Wi-Fi — WPA2; Zigbee — сетевой ключ (network key), не оставлять
  дефолтный; ESP-NOW — включать шифрование (PMK/LMK). Брокер MQTT — как в
  `docs/02_NETWORK.md` (внутренняя сеть, доступ узлов с LAN объекта).
- **Наблюдаемость:** router/шлюз должны слать `status online/offline` (как обычные
  узлы), чтобы «тишина» детектировалась нашим контуром.

## Вывод / рекомендация

1. **Сначала — вариант A** (сетевой Wi-Fi: репитер/второй AP/mesh-роутеры). В 90%
   «трудных» точек этого достаточно, и контур вообще не трогаем.
2. Для **реально радио-глухих** точек — **вариант B (Zigbee + Zigbee2MQTT → наш
   MQTT)** как честный стандартный mesh; ключевой объём работы — **маппинг топиков**
   в `monitoring/<node>/<metric>`, а не бэкенд.
3. **Вариант C (ESP-NOW)** — нишево, когда Zigbee избыточен.

Во всех случаях основной v1-стек (ESPHome + Wi-Fi + MQTT) остаётся, mesh — точечно
для конкретных зон. Переход на Zigbee/Thread где-либо в проде — **только отдельным
issue с обоснованием** (CLAUDE.md §6) после полевой проверки покрытия.
