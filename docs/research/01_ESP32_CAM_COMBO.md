# Research · Комбинированный узел «камера + датчики» на ESP32-CAM

> Статус: **research / гипотеза**, приоритет низкий (см. issue #234). Это не
> решение к внедрению, а проработка применимости. Цифры, помеченные
> «(проверить на железе)», требуют замера на реальной плате до любого пилота.

## 1. Идея и как она ложится в контур

Одно устройство **ESP32-CAM** (модуль ESP32 + камера OV2640) совмещает:
- **камеру** — отдаёт MJPEG-поток (ESPHome `esp32_camera`);
- **датчики среды** — SHT41/SHT40 (воздух) и/или MLX90614 (ИК-поверхность) по I²C;
- **публикацию** показаний в MQTT (как обычный узел датчиков).

**Бэкенд менять не нужно** — для нашего контура это просто:
- `sensor_node` (источник показаний по MQTT, см. `docs/08_MQTT_CONTRACT.md`);
- `camera` с MJPEG-URL (источник для go2rtc и для браузерного «Живого анализа»,
  см. `docs/07_VIDEO_ANALYTICS.md`).

То есть одно физическое устройство заводится в справочниках как **узел + камера**;
правок API/моделей данных не требуется.

## 2. Аппаратные ограничения ESP32-CAM (AI-Thinker)

| Параметр | Характеристика | Следствие |
|---|---|---|
| SoC | ESP32 (2 ядра, Wi-Fi b/g/n, BT); штатно 4 МБ PSRAM + 4 МБ flash (но встречаются клоны без PSRAM — **проверить плату**) | впритык хватает на стрим + сенсоры |
| Камера | OV2640, 2 МП, **rolling shutter**, **фикс-фокус**, без ИК-подсветки | см. §4 о качестве распознавания |
| Питание | стабильно 5 В; пики тока при Wi-Fi/стриме ориентировочно ~0.3–0.5 А (зависит от режима) | нужен нормальный БП, не «от платы программатора» |
| USB | **нет** — нужен USB-UART (FTDI) для прошивки | для серийной заливки неудобно |
| microSD | слот занимает GPIO 2,4,12,13,14,15 | **отключаем SD**, чтобы освободить GPIO под I²C |

### Дефицит GPIO — это главное

Камера занимает почти все выводы: шину данных D0–D7, `XCLK`, `PCLK`, `VSYNC`,
`HREF` и **SCCB** (служебная I²C камеры) на `GPIO26 (SIOD/SDA)` и
`GPIO27 (SIOC/SCL)`. SCCB — **отдельная** от пользовательской шина, вешать на неё
свои датчики нельзя.

Свободные выводы под **пользовательский I²C** появляются только если **отключить
microSD**. Тогда доступны `GPIO 2, 4, 12, 13, 14, 15` (с оговорками по
strapping-пинам). Рабочий вариант:
- **SDA = GPIO15, SCL = GPIO14** (либо 13/14). Важно: на ESP32 **strapping-пины
  есть и здесь** — `GPIO0` (boot), `GPIO2`, `GPIO12 (MTDI)` и **`GPIO15 (MTDO)`**.
  `GPIO12` **избегаем** (должен быть LOW в момент загрузки — подтяжка датчика
  сорвёт старт). `GPIO15` — MTDO, при сбросе должен быть HIGH; на AI-Thinker есть
  встроенный pull-up, а I²C тоже подтягивает к 3.3 В, так что обычно совместимо,
  но **проверить на железе** (иначе можно потерять boot-лог/режим).
- `GPIO4` — встроенный белый LED-«вспышка» (яркий), его лучше не занимать I²C.

Итог: **2 свободных вывода под I²C при выключенном SD** — этого хватает на шлейф
SHT4x + MLX90614 (оба на одной шине, разные адреса: SHT4x `0x44`, MLX90614 `0x5A`).

## 3. Прошивка ESPHome (эскиз, не финал)

ESPHome поддерживает `esp32_camera` и I²C-датчики одновременно. Набросок
конфигурации (распиновка AI-Thinker; **проверить на железе**):

```yaml
esphome:
  name: combo-node-01

esp32:
  board: esp32dev
  framework:
    type: esp-idf

psram:                     # OV2640 на 2 МП требует PSRAM

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password

mqtt:                      # показания датчиков → брокер контура
  broker: !secret mqtt_host
  topic_prefix: monitoring/combo-node-01

# --- Камера (распиновка AI-Thinker ESP32-CAM) ---
# SCCB камеры (GPIO26/27) — отдельная служебная шина, ею управляет esp32_camera.
# ВАЖНО: точные ключи SCCB и способ развести её с пользовательским I²C зависят от
# версии ESPHome. В новых версиях обе шины задаются блоками `i2c:` с разными `id`,
# а компоненты ссылаются на нужную через `i2c_id:`. Сверьтесь с докой вашей версии.
esp32_camera:
  name: combo-cam-01
  external_clock: { pin: GPIO0, frequency: 20MHz }
  data_pins: [GPIO5, GPIO18, GPIO19, GPIO21, GPIO36, GPIO39, GPIO34, GPIO35]
  vsync_pin: GPIO25
  href_pin: GPIO23
  pixel_clock_pin: GPIO22
  # SCCB камеры на GPIO26 (SDA)/GPIO27 (SCL) — задаётся ключами камеры/`i2c_id`
  # под вашу версию ESPHome (НЕ путать с пользовательской шиной ниже).
  resolution: 640x480       # компромисс качество/нагрузка (см. §4/§5)
  jpeg_quality: 12
  max_framerate: 5 fps      # стрим у esp32_camera и так невысокий

# Веб-сервер ESPHome отдаёт MJPEG-поток камеры (для go2rtc/просмотра)
esp32_camera_web_server:
  - port: 8081
    mode: stream

# --- Пользовательский I²C (только при ОТКЛЮЧЁННОМ microSD) ---
i2c:
  id: user_i2c
  sda: GPIO15
  scl: GPIO14
  frequency: 50kHz          # длинный шлейф — снижаем частоту

sensor:
  - platform: sht4x          # температура/влажность воздуха
    i2c_id: user_i2c
    address: 0x44
    temperature: { name: "Воздух · температура" }
    humidity: { name: "Воздух · влажность" }
    update_interval: 30s
  - platform: mlx90614       # бесконтактная ИК-температура поверхности
    i2c_id: user_i2c
    address: 0x5A
    object: { name: "Поверхность · ИК-температура" }   # ключ object, не object_temperature
    ambient: { name: "Около датчика · температура" }
    update_interval: 30s
```

Замечания:
- `esp32_camera_web_server` в режиме `stream` отдаёт **MJPEG**; go2rtc забирает
  его как HTTP-MJPEG-источник — **транскод не нужен** (кадры уже JPEG). Но поток
  обычно **одно-клиентный** и низкочастотный.
- Камера и пользовательский I²C сосуществуют, потому что у камеры **своя** шина
  SCCB (GPIO26/27), а датчики — на пользовательской шине (GPIO14/15). Развести их
  по разным `i2c`-`id` обязательно — иначе ESPHome выдаст конфликт шин.

## 4. Качество распознавания поз/протирки с OV2640

Наш пайплайн — MediaPipe `PoseLandmarker` (см. `docs/07_VIDEO_ANALYTICS.md`).
Ограничения матрицы OV2640 бьют именно по «тонким» сценариям:

- **Фикс-фокус** — резкость только в узком диапазоне дистанций; протирка стола
  вблизи может быть мыльной.
- **Rolling shutter** — при быстрых махах руками искажение геометрии кисти →
  «дрожание» landmark, ложные/пропущенные реверсы в детекторе протирки
  (детектор чувствителен к числу разворотов кисти, см. `analysis-core`
  `countReversals`).
- **Нет ИК / слабый сенсор в темноте** — при недостатке света шум растёт,
  стабильность скелета падает.
- **2 МП и низкий fps стрима** — для позы достаточно 480p–720p, но низкий fps
  (≤5) ухудшает детект **действий** (протирка/махи строятся на динамике окна
  кадров).

**Гипотеза по применимости (проверить на железе):**
- presence / грубые позы (рука поднята, наклон) — **скорее годится** при хорошем
  свете и дистанции 1.5–3 м;
- протирка стола / мелкая моторика — **на грани**; вероятно ненадёжно, особенно
  при слабом свете и быстрых движениях;
- основная видеоаналитика контроля — **нет**, нужны нормальные IP-камеры (H264,
  автофокус, нормальная матрица), как и заложено в `docs/01_ARCHITECTURE.md`.

## 5. Самонагрев и достоверность co-located датчика температуры

ESP32 + активная камера + Wi-Fi рассеивают порядка **0.5–1.5 Вт**, плата
ощутимо греется. Датчик температуры **воздуха (SHT4x), размещённый на самой
плате**, будет **завышать** показания на несколько градусов.

- Ожидаемое смещение: **+2…+8 °C** относительно реального воздуха
  (зависит от корпуса/обдува/fps — **измерить на железе**).
- **Митигация:** вынести SHT4x **на шлейф** (10–30 см) от платы, в стороне от
  потока тёплого воздуха; снизить fps/разрешение; не закрывать плату в глухой
  корпус. MLX90614 (меряет поверхность, а не воздух у платы) менее чувствителен,
  но его «ambient» канал — тоже нет.
- Для холодильных камер (где важна точность) co-located воздушный датчик на
  греющейся плате **неприемлем** — только вынос на шлейф или отдельный узел.

## 6. Нагрузка и стабильность

- Одновременно: MJPEG-стрим + опрос I²C + Wi-Fi. PSRAM обязателен; при 2 МП и
  высоком fps возможны **перезагрузки по нехватке памяти/таймаутам** — отсюда
  консервативные `640x480 @ ≤5 fps`.
- `esp32_camera` обычно отдаёт **один** клиентский поток; одновременный «живой
  анализ в браузере» **и** серверный воркер на одну ESP32-CAM могут конкурировать
  за единственный поток — для нашего контура источник один (go2rtc),
  а он мультиплексирует потребителей. **Проверить**, что go2rtc стабильно держит
  HTTP-MJPEG от ESPHome.
- Прошивка по UART (нет USB) — серийное обслуживание неудобно; для парка узлов
  закладывать **OTA** (ESPHome умеет).

## 7. Вывод

Комбо-узел на ESP32-CAM — **жизнеспособен для дешёвых/второстепенных точек и
пилота** (обзорное видео + базовая среда), при условиях:
1. microSD отключён, I²C на GPIO14/15, датчик воздуха **вынесен на шлейф**;
2. хороший свет, дистанция 1.5–3 м, ожидания по аналитике — presence/грубые позы;
3. питание 5 В от нормального БП, OTA для обслуживания.

**Не подходит** для основной видеоаналитики контроля и для точных измерений
температуры воздуха co-located датчиком. **Основной путь контура не меняем**
(IP-камеры H264 + отдельные узлы датчиков ESP32-C3, как в
`docs/01_ARCHITECTURE.md` и решении по датчикам).

**Дальнейшие шаги (если возникнет потребность):** собрать один прототип, замерить
(а) стабильность landmark на протирке при нашем свете, (б) смещение температуры
co-located vs вынесенного датчика, (в) стабильность go2rtc↔ESPHome MJPEG под
нагрузкой. Только после замеров — решать о пилоте.
