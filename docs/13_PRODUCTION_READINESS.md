# 13 · Готовность к продакшену

> Снимок на 2026-06-13 (ветка `main`). Назначение — честная оценка: что готово
> к выводу на объект, что заглушено намеренно (граница АУРА v1), какие реальные
> разрывы закрыть до боевого объекта, и как сделать развёртывание и заведение
> объекта удобным для инженера.

## 1. Вердикт

**Функциональное ядро v1 готово и проверено на реальном железе.** В этой сессии
прогнан полный тракт на 5 живых камерах: медиа (go2rtc), видеоаналитика (Python-
и Node-воркер), распознавание поз и эвристика «белого халата», журнал событий,
дашборды. Датчики проверены на демо-потоке и в контракте MQTT; узел собирается по
`docs/11_HARDWARE.md`.

**До боевого объекта остаются точечные доработки** (см. §4) — ни одна не
переписывает архитектуру: DELETE камеры, таймаут пробы живости, бэкап вне
сервера, политика retention. Развёртывание «почти одной командой» достижимо
небольшим `install.sh` (§5).

## 4. Реальные разрывы и риски (с приоритетом)

> Проверено по коду; не путать с заглушками АУРА.

### Высокий приоритет

1. **Нет удаления/смены помещения у камер, узлов, помещений.**
   У `cameras`/`sensor_nodes`/`rooms` есть только create + (у камеры) PATCH
   тумблеров — **DELETE отсутствует, помещение камеры сменить нельзя**. Ошибся
   при заведении → правится только напрямую в БД. (У зон, порогов, правил
   уборки/присутствия, расписаний DELETE **есть** — там всё в порядке.)
   *Фикс:* `DELETE /cameras/{id}` (+ каскад зон) или soft-delete; то же для
   узлов/помещений. ~1 день.

2. **Таймаут пробы живости камер 3 с → ложные `camera_offline`.**
   `Go2rtcCameraProber(timeout=3.0)`. go2rtc подключается к камере по требованию;
   «холодный» первый кадр RTSP часто >3 с → проба ложно таймаутит. В полевом
   тесте все простаивающие камеры мигали offline, «online» только во время
   анализа. *Фикс:* вынести таймаут в env (`CAMERA_PROBE_TIMEOUT_S`, ~8 с) и
   не дёргать все камеры в один тик. ~0.5 дня.

3. **Бэкап БД лежит на том же сервере.** `backup` пишет дампы в том `backups`
   на том же диске — отказ диска/шифровальщик уносит и БД, и дампы. *Фикс:*
   push-копия на внешний носитель (S3/SMB/rsync) или явный пункт в runbook.

### Средний приоритет

4. **Retention сырых показаний (#41) — `READINGS_RETENTION_DAYS=0` (выключен).**
   Нужен регламент заказчика; иначе БД растёт. До объекта — решить и
   задокументировать.

5. **Медиа-ключ в URL** (`?api_key=` для `<img>`/потока) — ключ виден в HTML
   сессии. v1-смягчение: GUI держать во внутренней сети. Полноценно — #257 (v2).

6. **Нет rate-limit на API.** Для объекта во внутренней сети некритично; при
   публикации наружу — добавить.

### Решение, а не разрыв

7. **Публичный репозиторий.** Сделан публичным осознанно (бесплатные минуты CI).
   **История git просканирована — секретов нет** (`.env`/`secrets.yaml`/`*.key`
   никогда не коммитились, реальные адреса камер не в истории, в `.env.example`
   только плейсхолдеры). Это **не утечка**, а бизнес-решение: публичны материалы
   заказчика (`docs/customer/`, дизайн интеграции с АУРА). Если по договору это
   нежелательно — вернуть в приватный (тогда для CI поднять spending limit или
   вынести customer-доки). Видимость обратима.

## 5. Развёртывание «одной командой» на свежей Ubuntu

**Сейчас — почти, но не совсем.** `scripts/bootstrap.sh` уже делает многое
идемпотентно: генерит `.env` со случайными секретами (`init_env.py`), создаёт
`go2rtc.yaml` из примера, тянет модель и веб-ассеты, применяет справочники
(`seed`), поднимает стек в правильном порядке (`depends_on` + healthcheck).

**Чего не хватает на голой Ubuntu:** на чистой системе нет Docker, docker compose
v2, Python 3, git — `bootstrap.sh` упадёт на первой же команде. Это
**предпосылки хоста**, а не логика контура.

**Решение — тонкий `scripts/install.sh` (одноразовый, под root):**

```bash
#!/usr/bin/env bash
set -euo pipefail
# 1) системные зависимости
apt-get update
apt-get install -y --no-install-recommends \
  ca-certificates curl git python3 python3-venv
# 2) Docker + compose v2 (официальный скрипт Docker ставит и плагин compose)
curl -fsSL https://get.docker.com | sh
systemctl enable --now docker
# 3) код и запуск
git clone <repo-url> monitoring && cd monitoring
scripts/bootstrap.sh --demo      # или без --demo для боевого режима
```

После этого **демо реально поднимается одной командой** (`bootstrap.sh --demo`).
Боевой режим — та же команда без `--demo`, плюс **ручные ассеты объекта** (их
автоматизировать нельзя — они уникальны):

- `.env` секреты — генерятся, но боевые пароли стоит задать осознанно;
- `media-gateway/go2rtc.yaml` — RTSP-адреса камер (см. §7);
- `db/seeds/object.yaml` — помещения/узлы/камеры/пороги (или через GUI);
- модель MediaPipe — тянется автоматически при наличии сети.

**Вывод:** «одна команда на свежей Ubuntu» достижима для **демо** прямо сейчас
небольшим `install.sh`; для **боевого** — «одна команда + заполнить ассеты
объекта», что неизбежно. Реализация `install.sh` — отдельная задача (~0.5 дня).

## 6. Удобное заведение объекта (помещения · узлы · камеры)

**Три способа, дополняют друг друга:**

| Способ | Когда удобен | Что умеет |
|---|---|---|
| **GUI** (`/ui/`, вкладка «Объект») | штучно, на месте, без CLI | помещения, узлы, камеры, пороги, правила, расписания — формами |
| **REST** (`/api/v1/...`) | скриптом/из АУРА (v2) | то же, программно; конверт ответа единый |
| **Сид-YAML** (`db/seeds/object.yaml` + профиль `seed`) | массово, «из коробки» | UPSERT помещений/узлов/камер **и порогов** одним файлом до старта |

**Точки трения (что улучшить):**

- **Нет DELETE и смены помещения** у камер/узлов/помещений (см. §4.1) — главный
  риск ошибки при заведении.
- **Связки, которые легко перепутать:** `node_id` прошивки = `sensor_nodes.id`;
  имя камеры = имя потока в `go2rtc.yaml`. При расхождении данные молча
  отбрасываются. *Улучшение:* предупреждать в GUI при несовпадении (камера без
  одноимённого потока в go2rtc / узел без прошивки).

**Идеальный быстрый поток «с нуля»:** один файл объекта `db/seeds/object.yaml`
(помещения + узлы + камеры + пороги) → `bootstrap.sh` применяет его при первом
старте → инженер только заполняет YAML по образцу `object.example.yaml`. Для
большого объекта — генератор YAML из CSV (см. §7 sensors). Это уже на 90%
реализовано; не хватает только DELETE для исправления ошибок и валидации связок.

## 7. Камеры: вендорское приложение + наш контур

**Камеры настраиваются в приложении вендора** — это правильно и не наша зона:
IP-адрес, RTSP-поток и его путь, разрешение/битрейт, логин/пароль, кодек
(нужен **H.264** — браузеры не играют H.265). Наш контур к **готовому** RTSP
только подключается.

**Что из вендора нужно нам:** ровно RTSP-URL потока (`rtsp://логин:пароль@IP:554/path`).
Он прописывается **один раз** в `media-gateway/go2rtc.yaml`:

```yaml
streams:
  cam-01:
    - rtsp://LOGIN:PASSWORD@192.168.88.5:554/1/1
    - ffmpeg:cam-01#video=mjpeg   # MJPEG-транскод для браузерного превью
```

**Ключевая связка:** имя потока в `go2rtc.yaml` (`cam-01`) обязано совпадать с
`name` камеры в справочнике — по нему GUI берёт кадр и воркер тянет
ретранслятор `rtsp://media-gateway:8554/cam-01`. go2rtc снимает с камеры один
RTSP-поток и раздаёт всем потребителям (превью, анализ, проба живости) — это
снимает лимит «1–2 подключения» дешёвых камер.

**Текущий поток подключения камеры:**
1. В приложении вендора задать IP/поток/разрешение/H.264, узнать RTSP-URL.
2. Добавить поток в `go2rtc.yaml` (имя = будущее имя камеры).
3. Завести камеру в справочнике (GUI/REST/seed) с тем же `name` и помещением.
4. Проверить кадр: GUI → «Загрузить кадр» (или обзорный экран).

**Точки трения и улучшения:**
- Адрес камеры дублируется: в `go2rtc.yaml` (с паролем) и в справочнике
  (`rtsp_url`). *Улучшение:* один источник истины — генерировать `go2rtc.yaml`
  из справочника камер (или наоборот), чтобы не расходились.
- Имя камеры ↔ имя потока — ручная дисциплина. *Улучшение:* при заведении
  камеры в GUI проверять, что одноимённый поток в go2rtc отвечает кадром.

## 8. Быстрая конфигурация датчиков

**От платы до показаний** (подробно — `docs/11_HARDWARE.md`):
1. Прошить узел: `pip install esphome` → `cp node.example.yaml node-XX.yaml`
   → поправить `substitutions` → `esphome run` (1-й раз по USB, далее OTA).
2. Завести узел в справочнике (`node_id` = `sensor_nodes.id`).
3. Задать пороги — и они **применяются на лету**.

**Пороги — сильная сторона: горячая перезагрузка.** Меняешь порог в GUI/REST →
`ingest-sensors` перечитывает из БД на каждом тике (~минута) → новый порог
действует со следующего показания. Перепрошивка/рестарт не нужны. Порог
«тишины» узла (`silent_min`) — там же, берётся самый строгий применимый.

**Что замедляет (и как ускорить):**
- **По файлу прошивки на узел.** Для большого объекта — генератор YAML из
  таблицы (`node_id, room, has_uv, i2c_freq`) на базе `node.example.yaml`.
- **Пороги пачкой** — сейчас по одному (GUI/REST) или массово через
  `object.yaml` (секция `thresholds`). Для REST удобен был бы `POST
  /thresholds` пачкой; пока массовый путь — сид-YAML.
- **Калибровка УФ-C и эмиссивность ИК** — ручные (см. `docs/11 §7`); для
  стандартных модулей завести таблицу коэффициентов.

**Идеальный поток:** один `object.yaml` описывает помещения + узлы (+ шаблон
прошивки) + пороги по типам помещений (cold/kitchen) → генератор делает и
конфиги прошивок, и сид БД. Группы порогов вместо построчного ввода.

## 9. Чек-лист перед выводом на объект

**Хост/деплой**
- [ ] Docker + compose v2, Python 3 (через `install.sh` или вручную).
- [ ] `.env` с боевыми секретами (не `change-me`); проверить `docker compose config`.
- [ ] Модель `models/pose_landmarker.task` на месте.
- [ ] Диск: ≥50 ГБ под БД + артефакты + бэкапы; NTP синхронизирован; `PRESENCE_TZ` объекта.

**Объект**
- [ ] Камеры настроены в вендорском приложении (H.264), RTSP-URL в `go2rtc.yaml`.
- [ ] Имена потоков go2rtc = имена камер в справочнике.
- [ ] Помещения/узлы/камеры заведены (seed или GUI); `node_id` = id узлов прошивок.
- [ ] Пороги заданы (хотя бы для холодильных камер, со `silent_min`).

**Проверка**
- [ ] `docker compose ps` — все healthy; `/api/v1/health` 200.
- [ ] Grafana поднимает datasource; обзорный экран показывает помещения/камеры.
- [ ] Тестовый бэкап снят и проверено восстановление на отдельном томе.

**Решения с заказчиком**
- [ ] Политика retention сырых показаний (#41).
- [ ] Куда копировать бэкапы вне сервера.
- [ ] Приватность репозитория (если публичность нежелательна).

## 10. Приоритет доработок до боевого объекта

1. `DELETE /cameras/{id}` (+ узлы/помещения) — убрать риск необратимой ошибки. **~1 д**
2. `CAMERA_PROBE_TIMEOUT_S` (≈8 с) — убрать ложные `camera_offline`. **~0.5 д**
3. `scripts/install.sh` — «одна команда» на свежей Ubuntu (демо). **~0.5 д**
4. Push-копия бэкапа вне сервера (или явный runbook). **~1 д**
5. Регламент retention (#41) — обсуждение с заказчиком, затем UI/доки.
6. Валидация связок камера↔поток и узел↔прошивка в GUI. **~0.5 д**

v2 (по плану): интеграция АУРА, медиа-токены (#257), полное переключение на
Node-воркер (#319, полевая проверка пройдена).
