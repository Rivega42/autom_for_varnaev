# 06 · План работ (эпики → атомарные задачи → issue)

Здесь — разбивка на эпики и атомарные задачи. **Каждая задача = отдельный issue.**
Эпик = milestone (или метка `epic:E#`). Заводи issue по этому плану; если задача
кажется крупной — дроби дальше (критерии атомарности — `00_BEST_PRACTICES_CODE.md`).

Порядок эпиков — это и есть рекомендуемый порядок работы (E0 → E9).

---

## Шаблон issue

> Используй этот шаблон для **каждого** issue. Всё по-русски, кроме идентификаторов.

```
## Задача
<одно предложение: сделать X, чтобы стало Y>

## Контекст
<ссылки на документы: docs/01_ARCHITECTURE.md §..., и т.п.>

## Критерии приёмки (Definition of Done)
- [ ] <проверяемый результат 1>
- [ ] <проверяемый результат 2>
- [ ] тест(ы) добавлены и проходят
- [ ] ruff + mypy чистые
- [ ] если менялся контракт — обновлён docs/<файл>

## Затрагиваемые файлы/каталоги
<services/..., db/..., shared/...>

## Зависит от
#<номер issue> (если есть)

## Метки
epic:E# · type:* · area:* · prio:* [· stub-aura]
```

---

## Метки (labels)

- Эпик: `epic:E0` … `epic:E9`
- Тип: `type:feat` `type:fix` `type:docs` `type:infra` `type:test`
- Область: `area:sensors` `area:analytics` `area:log` `area:api` `area:db` `area:infra` `area:docs`
- Приоритет: `prio:high` `prio:med` `prio:low`
- Спец.: `stub-aura` (создаёт заглушённый разъём для v2)

---

## E0 · Каркас репозитория и инфраструктура

Цель: репо разворачивается и поднимается одной командой; есть линт/типы/тесты/CI.

Атомарные задачи (по одному issue на пункт):
1. Инициализировать монорепо по структуре `01_ARCHITECTURE.md` §8 (пустые каталоги сервисов + README-заглушки).
2. Добавить `.gitignore` (Python, Docker, `.env`).
3. Добавить `.env.example` со всеми переменными (без значений-секретов).
4. Настроить `ruff` (lint+format) — единый конфиг в корне.
5. Настроить `mypy` — единый конфиг.
6. Настроить `pytest` — корневой конфиг, прогон по всем сервисам.
7. Завести `shared/`: пакет общих моделей (пока пустой каркас + `pyproject`).
8. Каркас `docker-compose.yml`: только `db` и сети `internal`/`integration`, том `artifacts`.
9. Поднять `db` (TimescaleDB) в compose, healthcheck.
10. CI: workflow, прогоняющий ruff + mypy + pytest на PR.
11. CI: сборка Docker-образов сервисов (smoke-build).
12. `README.md` репозитория: как поднять локально (dev-инструкция).

## E2 · Контур датчиков

Цель: показания доходят от MQTT до БД, пороги дают события. **Принцип:** показания
в v1 в АУРА не передаются — контур выводит из них события по критериям (симметрично
видео, см. `01_ARCHITECTURE.md` §5). Каждое событие несёт человекочитаемое
сообщение `message` (RU, с контекстом помещения).
1. Поднять `mqtt-broker` (Mosquitto) в compose (только сеть `internal` + публикация на LAN).
2. Определить формат MQTT-топиков и payload ESPHome (документировать в `docs`).
3. `ingest-sensors`: каркас воркера, подключение к брокеру.
4. `ingest-sensors`: подписка на топик показаний.
5. `ingest-sensors`: парсинг payload → модель `Reading` (+ тест парсинга).
6. `ingest-sensors`: запись показания в `sensor_readings`.
7. `ingest-sensors`: загрузка `thresholds` и сверка показаний.
8. `ingest-sensors`: генерация события `threshold_exceeded` (с `message` на русском) → `log-service` (внутр. REST).
9. `ingest-sensors`: генерация `back_to_normal` (с `message`).
10. `ingest-sensors`: контроль «тишины» узла → событие `sensor_silent` (с `message`).
11. `firmware/esphome`: эталонный YAML узла по решению проекта — **ESP32-C3 (SuperMini) + SHT41/SHT40 (или BME280) + MLX90614/GY-906** на I²C, Wi-Fi, deep_sleep, OTA.
12. `firmware/esphome`: вариант для холодильной камеры (контроллер с радио/питанием снаружи, SHT4x+MLX90614 внутри на I²C-шлейфе через уплотнитель) — конфиг + заметка по монтажу и калибровке ИК (эмиссивность ε≈0,95 для матовых; по блестящему металлу — матировать/поправка).
13. Интеграционный тест: эмулятор MQTT-сообщения → запись в БД появилась.

## E3 · Лог-сервис событий

Цель: единый журнал принимает события и отдаёт их по REST.
1. `log-service`: каркас FastAPI, подключение к БД.
2. `log-service`: внутренний эндпойнт приёма события (от ingest и analytics).
3. `log-service`: запись события в `events` (+ связь с `artifacts`/`analysis_tasks`).
4. `log-service`: валидация типов событий по `04_DATA_MODEL.md` (+ тест).
5. `log-service`: чтение `GET /events` с фильтрами (время/тип/помещение).
6. `log-service`: чтение `GET /events/{id}`.

## E4 · Видеоаналитика — **перенос рабочего PoC** (не пишем заново)

Цель: перенести проверенный движок из браузерного PoC (`motion-log.html`) в
Python-воркер. Эталон поведения и формулы — `docs/07_VIDEO_ANALYTICS.md`.
**Сначала** положить PoC в репозиторий как референс, **затем** портировать по
частям, сверяя поведение с оригиналом.

1. Положить исходный `motion-log.html` в `services/video-analytics/reference/` (источник логики для порта).
2. `media-gateway`: поднять go2rtc в compose (сеть `internal`), конфиг приёма RTSP/ONVIF и раздачи воркеру + превью WebRTC.
3. `video-analytics`: каркас воркера, чтение задания из `analysis_tasks`.
4. `video-analytics`: абстракция «источник» — интерфейс `stream` (через media-gateway) | `file` (путь на томе).
5. `video-analytics`: источник `stream` (получение кадров от media-gateway, понижение до 5–8 fps).
6. `video-analytics`: источник `file` — режим рабочий, по триггеру `# СТЫК-АУРА (v2)`.
7. `video-analytics`: поднять MediaPipe `PoseLandmarker` (Python Tasks), пайплайн `pose_v1`; зеркалирование/анатомические лево-право как в PoC (+ тест на фикстуре кадров).
8. `video-analytics`: порт **простых событий** (рука/колено/голова/корпус) — пороги с гистерезисом и антидребезгом → события `pose_event` (+ тест).
9. `video-analytics`: порт **составных действий** (протирание двумя руками, махи, хлопок, ходьба) — скользящее окно → события `action_detected` с длительностью (+ тест).
10. `video-analytics`: порт **ROI-зон** — чтение `camera_zones`, расчёт **% покрытия** по типам зон → события `coverage_report`.
11. `video-analytics`: порт **эвристики «белого халата»** (торс по точкам 11/12 и 23/24, яркость+насыщенность) → `condition_flagged`; флаг как индикатор/условие.
12. `video-analytics`: сохранение артефактов — скриншот-кадр (`screenshot`) и при необходимости keypoints/coverage (`keypoints`/`coverage`) на общий том по схеме пути.
13. `video-analytics`: отправка событий в `log-service` со ссылкой на артефакт.
14. `video-analytics`: обновление статуса задания (`running`/`done`/`failed`).
15. (v2, отдельный issue, `stub-aura`) обученный детектор формы/СИЗ (YOLO-класса) вместо эвристики — НЕ в v1.

> Что НЕ тащим в v1 (см. `07` §4): очередь (Kafka/Redis/NATS), GPU-батчинг,
> S3/MinIO. На одном сервере с немногими камерами — CPU, прямой REST в лог,
> артефакты на томе. Масштабирование — отдельное решение при росте числа камер.

## E6 · api-gateway (REST + разъёмы АУРА)

Цель: единственный внешний вход; рабочие эндпойнты v1 + заглушки v2.
1. `api-gateway`: каркас FastAPI, healthcheck, единый конверт ответа (из `shared/`).
2. `api-gateway`: коды ошибок и обработчик исключений → конверт.
3. `api-gateway`: `GET /events`, `GET /events/{id}` (проксирование к данным).
4. `api-gateway`: `POST /analysis-tasks`, `GET /analysis-tasks/{id}`, список.
5. `api-gateway`: `GET /readings`.
6. `api-gateway`: фичефлаг `aura_integration_enabled` (по умолчанию false).
7. `api-gateway`: `/integration/analysis-tasks` → 501 при выключенном флаге · `stub-aura`.
8. `api-gateway`: `/integration/events` → 501 · `stub-aura`.
9. `api-gateway`: `/integration/settings` → 501 · `stub-aura`.
10. `api-gateway`: аутентификация по `X-API-Key` на публичных и `/integration/*`.
