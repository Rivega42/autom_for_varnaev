# integration — артефакты интеграции с АУРА

Машиночитаемое описание интеграционной поверхности контура (разъёмы для **АУРА**),
для разработчиков АУРА. Источник истины и подробные примеры — контракт
[`docs/03_API_CONTRACT.md`](../docs/03_API_CONTRACT.md) §4 и
[`docs/customer/РУКОВОДСТВО_ЗАКАЗЧИКА.md`](../docs/customer/РУКОВОДСТВО_ЗАКАЗЧИКА.md)
Приложение D (сценарии D.1–D.5).

## Файлы

- **`aura_integration.openapi.yaml`** — OpenAPI 3.1-фрагмент интеграционной
  поверхности: разъёмы `/integration/*`, опрос задания `/analysis-tasks/{id}` и
  обратный вызов готовности (webhook). Для кодогенерации клиента и валидации.
- **`aura_integration.postman_collection.json`** — Postman-коллекция (v2.1):
  импортируйте, задайте переменные `base_url` и `api_key`, слать запросы D.1–D.5.

## Быстрый старт (Postman)

1. Импортировать `aura_integration.postman_collection.json`.
2. В переменных коллекции задать:
   - `base_url` — адрес шлюза контура (напр. `http://<сервер>:8000/api/v1`);
   - `api_key` — значение `API_KEY` из `.env` контура;
   - при отладке D.5 — `callback_url` на свой приёмник.
3. Слать запросы D.1 → D.2 → D.3 → D.4; D.5 — пример уведомления **наш контур →
   АУРА** (приёмник реализует АУРА).

## Граница версий (важно)

В **версии 1** все `/integration/*` отвечают **HTTP 501**
(`error.code = NOT_IMPLEMENTED`), пока выключен фичефлаг
`AURA_INTEGRATION_ENABLED`. **Адреса и тела уже стабильны** — клиента можно
генерировать заранее; в **версии 2** те же эндпойнты отвечают `200`.

> Эти артефакты держатся в синхроне с кодом: тест `tests/test_integration_artifacts.py`
> сверяет пути OpenAPI-фрагмента с реально зарегистрированными разъёмами шлюза.
