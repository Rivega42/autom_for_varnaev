# CLAUDE.md — постоянные инструкции для Claude Code

> Этот файл читается тобой (Claude Code) в начале каждой сессии и задаёт правила,
> которые действуют для **всего** репозитория и **всех** задач. Если какая-то
> конкретная задача (issue) противоречит этому файлу — приоритет у этого файла,
> а противоречие нужно вынести в комментарий к issue, а не решать молча.

## 1. Что мы строим (в двух абзацах)

Комплекс мониторинга помещений на объекте: в каждом помещении стоят датчики
(температура воздуха, влажность, бесконтактная ИК-температура поверхности) и
камеры. Все данные сводятся на **один физический сервер на объекте**. На сервере
в контейнерах работает **наше ПО** (приём и накопление данных датчиков,
видеоаналитика, единый журнал событий) рядом с продуктом заказчика **АУРА**
(контроль производства, тех. карты, программа производственного контроля,
видеозапись).

**Мы делаем только наш контур.** АУРА — продукт заказчика, мы её не пишем и не
трогаем. В версии 1 наш контур работает как **standalone**: он не вызывает АУРА и
не вызывается ею, но архитектурно готов к будущему стыку (см. §4).

**Два блока опираются на уже сделанную работу — продолжаем, а не переделываем:**
- **Видеоаналитика** переносится из рабочего браузерного PoC (`motion-log.html`):
  MediaPipe `PoseLandmarker`, события/действия, ROI-зоны, % покрытия, эвристика
  «белого халата». Эталон поведения — `docs/07_VIDEO_ANALYTICS.md`. Это **порт**,
  а не разработка с нуля.
- **Датчики** собраны по решению проекта (файл «Мониторинг_среды_решение.md»):
  ESP32-C3 + SHT41/SHT40 + MLX90614 на I²C, ESPHome, MQTT, холодильные камеры на
  I²C-шлейфе. Стек не пересматриваем, продолжаем с него.

## 2. Язык и оформление (обязательно)

- **Вся документация — на русском.** README, доки в `docs/`, описания issue,
  тексты в Pull Request, сообщения коммитов (тело) — по-русски.
- **Все комментарии в коде — на русском.** Docstring'и функций, поясняющие
  комментарии, TODO — по-русски.
- **Идентификаторы в коде** (имена переменных, функций, классов, эндпойнтов,
  полей JSON, таблиц БД) — на **английском** по общепринятым конвенциям. Это не
  переводим: `sensor_reading`, а не `pokazanie_datchika`.
- **Сообщения логов приложения** — на русском, если они адресованы оператору;
  технические debug-логи — допускается английский.
- Кодировка везде UTF-8. Перевод строк — LF.

## 3. Как мы дробим работу (очень важно)

- Работа разбита на **эпики** (крупные блоки, см. `docs/06_BUILD_PLAN.md`).
- Каждый эпик дробится на **атомарные задачи**. Атомарная — значит: одна задача =
  одно осмысленное изменение, которое можно сделать и проверить независимо,
  за обозримое время, и которое не тянет за собой «заодно ещё пять дел».
  Если задачу нельзя описать одним предложением «сделать X, чтобы стало Y» —
  она не атомарна, её надо разбить дальше.
- **На каждую задачу — свой отдельный issue.** Не объединяй несколько задач в
  один issue. Не делай «issue-зонтик с чеклистом на 15 пунктов» — вместо этого
  заведи 15 issue и свяжи их с эпиком (через milestone или метку эпика).
- **Один issue → один Pull Request → один merge.** PR закрывает ровно один issue.
- Если по ходу работы над issue ты обнаружил ещё работу — **не делай её в этом
  же PR**. Заведи новый issue и продолжай.

Шаблон issue и метки — в `docs/06_BUILD_PLAN.md`, раздел «Шаблон issue».

## 4. Граница v1 / v2 — что «живое», а что «заглушка»

Это критично. Не реализуй интеграцию с АУРА в v1.

| Что | v1 (сейчас) | v2 (потом) |
|---|---|---|
| Наш контур (датчики, аналитика, лог) | **полностью рабочий** | то же + интеграция |
| REST-эндпойнты для АУРА (`/integration/...`) | **существуют, но заглушены** (возвращают `501 Not Implemented` или фичефлаг `aura_integration_enabled=false`) | включаются |
| Триггер аналитики | по расписанию (наш планировщик) | + по команде АУРА (REST) |
| Источник видео для аналитики | RTSP-поток | RTSP-поток ИЛИ видеофайл/фрагмент от АУРА |
| Данные датчиков | показания **только в нашем контуре** (БД); наружу — **события**, не сырые ряды | АУРА читает **события** датчиков (не показания) |
| Сеть `integration` (Docker) | **создаётся**, но обмена нет | по ней идёт обмен с АУРА |

**Правило:** всё, что касается АУРА, должно быть в коде *предусмотрено* (разъём,
эндпойнт, параметр-источник), но *не активировано*. Активация — через
конфиг/фичефлаг, а не через дописывание кода. Помечай такие места комментарием
`# СТЫК-АУРА (v2): ...`.

**Симметрия датчиков и видео (важно).** Наш контур наружу отдаёт **события**, а не
сырьё. Как видеоаналитика выдаёт распознанные события (а не сырые кадры), так и
подсистема датчиков выдаёт события, выведенные из показаний **по критериям**
(пороги, «тишина» узла), а **не сами ряды**. В v1 показания датчиков в АУРА не
передаются вовсе (интеграция заглушена); они копятся в нашей БД и доступны нам
(Grafana, `GET /readings`). Каждое событие несёт человекочитаемое сообщение для
оператора (`events.message`, по-русски) — например «В холодильной камере
температура выше нормы» или «В помещении для приготовления пищи влажность
вернулась к норме». См. `docs/04_DATA_MODEL.md` §4 и `docs/03_API_CONTRACT.md` §3.2.

## 5. Что НЕ делать (границы ответственности)

- **Не реализуй видеозапись** — это функция АУРА. Мы только аналитика
  (real-time или по файлу) и скриншоты-доказательства.
- **Не реализуй тех. карты, контроль производства, ППК** — это АУРА.
- **Не пиши код, обращающийся к внутренним сервисам АУРА.** В v1 их нет, в v2
  общение только через наш `api-gateway` по REST.
- **Не выставляй наружу** (за пределы внутренней Docker-сети) ничего, кроме
  `api-gateway` и дашборда. БД, брокер, лог-сервис, аналитика — только во
  внутренней сети. См. `docs/02_NETWORK.md`.
- **Не передавай сырые показания датчиков в АУРА в v1.** Наружу контур отдаёт
  только **события** (выведенные по критериям), а не временные ряды — симметрично
  видео (см. §4, блок «Симметрия датчиков и видео»).
- **Не клади секреты в репозиторий.** Пароли, токены, ключи — через `.env`
  (в `.gitignore`) и переменные окружения. В репо — только `.env.example`.

## 6. Технологический стек (зафиксирован, не меняем без issue с обоснованием)

- Язык сервисов: **Python 3.12**, фреймворк REST — **FastAPI**.
- Видеоаналитика: **Python + MediaPipe Tasks `PoseLandmarker`** (порт PoC `motion-log.html`).
- Медиа-шлюз камер: **go2rtc** (альт. MediaMTX) — RTSP/ONVIF → воркер + превью.
- БД: **TimescaleDB** (PostgreSQL 16 + расширение timescaledb).
- Брокер датчиков: **Mosquitto (MQTT)**.
- Прошивка узлов датчиков: **ESPHome** (YAML, без кода) на ESP32-C3.
- Дашборды: **Grafana** поверх TimescaleDB.
- Контейнеризация: **Docker + docker-compose**.
- Миграции БД: **Alembic** (или goose, если решим иначе — но тогда issue).
- Тесты: **pytest**.
- Линт/формат: **ruff** (lint+format) + **mypy** (типы).

Полное обоснование — `docs/01_ARCHITECTURE.md`.

## 7. Конвенции кода и репозитория

- **Структура репозитория** задана в `docs/01_ARCHITECTURE.md` §«Структура
  репозитория». Следуй ей, не придумывай свою.
- **Коммиты — Conventional Commits**: `feat:`, `fix:`, `docs:`, `chore:`,
  `test:`, `refactor:`. Заголовок можно по-английски (тип), тело — по-русски.
  В конце тела ссылка на issue: `Closes #N`.
- **Каждая задача с кодом сопровождается тестом.** Нет теста — задача не закрыта.
- **Каждое изменение контракта** (REST, схема события, схема БД) **обновляет
  соответствующий документ** в `docs/` в том же PR. Документы — источник истины.
- **api-gateway** обязан отдавать OpenAPI (FastAPI делает это сам по
  `/openapi.json` и `/docs`) — это машинно-читаемая версия `docs/03_API_CONTRACT.md`.
- Единый формат ответа API (конверт) — см. `docs/03_API_CONTRACT.md`. Не
  изобретай свой формат на каждом эндпойнте.

## 8. Определение готовности (Definition of Done) для задачи

Задача (issue) считается выполненной, когда:

1. Код написан, комментарии на русском.
2. Есть тест(ы), они проходят локально.
3. `ruff` и `mypy` не ругаются.
4. Если менялся контракт — обновлён соответствующий `docs/`-файл.
5. PR ссылается на issue (`Closes #N`), описание PR на русском.
6. Если задача затрагивала границу АУРА — соответствующий код помечен
   `# СТЫК-АУРА (v2)` и оставлен заглушённым.

## 9. Порядок работы (рекомендуемый ритм с Claude Code)

1. Начинаешь эпик → сначала **планируешь** (режим plan): предлагаешь разбивку на
   атомарные задачи, заводишь issue, ждёшь подтверждения, **потом** пишешь код.
2. Берёшь один issue → ветка `feat/<issue-номер>-краткое-имя`.
3. Пишешь код + тест → прогоняешь линт/типы/тесты.
4. Открываешь PR, `Closes #N`, описание по-русски.
5. Один issue закрыт — берёшь следующий. Не копи незакрытые ветки.

Подробнее — `docs/00_BEST_PRACTICES_CODE.md`.
